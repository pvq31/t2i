# 修改记录 5.12

## 一、适合汇报的详细说明

本次修改把原本“LLM 生成 pkl、LLM 评估、LLM 按自然语言 guidance 修复”的流程，升级为一个具备 harness 特征的确定性闭环。核心变化是：空间关系、对象数量、尺寸体积、相机范围不再依赖 prompt 记忆，而是被转换为可计算、可验证、可修复的 pkl 参数约束。

新增的核心模块是 `harness_param_constraints.py`。它负责从场景文字和主体清单中构建 `object_manifest`，抽取空间谓词 `predicates`，再把谓词投影到 `subjects_data[*].dims/x/y/z/azimuth` 与 `camera_data` 的具体字段上。它会输出结构化的 `validation.json`、`repair_plan.json` 和 `repair_actions.jsonl`，使后续修复不再需要让模型“再试一次”，而是直接执行字段级修复动作。

当前四个 agent 的职责也被重新整理：

- `agent_text2pkl_v5.py`：生成初始 pkl 后立即运行 deterministic harness，保存 pkl 以及 `object_manifest/predicates/constraints/validation/repair_plan/repair_actions`。
- `agent_opinion.py`：在原有 A-F 评估基础上，新增 harness 机器验证，并可输出 `--repair-plan-output`。
- `agent_reverse.py`：新增 `--repair-plan` 模式，可以直接执行 JSON 修复计划，而不是只依赖自然语言 guidance。
- `run_scene_harness.py`：新增统一入口，实现计划、执行、验证、失败恢复/再执行的自动循环，只有 pkl 验证通过才进入 `infer2.py`。

此外，新增 `docs/harness/PROJECT_BOARD.md`、`docs/harness/FAILURE_KB.jsonl` 和 `docs/harness/MEMORY_POLICY.md`，用于保存项目看板、历史失败模式和上下文管理策略，满足跨对话记忆和 memory 管理需求。

补充新增两份 harness 规则文档，不是工具脚本：

- `docs/harness/LAYOUT_READABLE_DIMENSION_RULES.md`：定义 `inference/asset_dimensions.json` 作为 harness 布局可读 cube 尺寸表的规则，明确尺寸按 `width * depth * height` 的 cube 体积判断相对大小，并约束功能关系、体积排序和验收标准。
- `docs/harness/PROMPT_RELATION_RULES.md`：定义中英文 prompt 短语到 `centered/left_of/right_of/in_front_of/behind/above/below/support` 的硬规则映射，要求用户 prompt 使用文档中的规范说法，避免依赖自由语义理解导致误解析。

## 二、关键代码辅助介绍

### 1. 确定性约束工具

`harness_param_constraints.py` 提供统一接口：

```python
def run_harness_on_scene(
    scene_dict,
    scene_text,
    allowed_types,
    reference_dims_map,
    objects_spec="",
    object_manifest_path="",
    on_ambiguous="best_effort",
):
    manifest = build_object_manifest(...)
    predicates_payload = extract_predicates(scene_text, manifest)
    repaired_scene, repair_actions = apply_full_harness(...)
    validation = validate_scene(...)
    return {
        "scene": repaired_scene,
        "object_manifest": manifest,
        "predicates": predicates_payload,
        "constraints": {"actions": repair_actions},
        "validation": validation,
        "repair_plan": build_repair_plan(...),
    }
```

### 2. 空间关系转 pkl 字段约束

例如 `right_of(A, B)` 不再由 prompt 约束，而是直接修改和验证 `y[0]`：

```python
target_y = second_y + second_width / 2.0 + first_width / 2.0 + margin_y(first, second)
set_field_action(first, "y", max(subject_field(first, "y"), target_y), reason + ":y_lower", actions)
```

`support(A, B)` 会同时约束 `x/y/z` 和 footprint：

```python
set_field_action(subject, "z", sz + support_height, reason + ":support_z", actions)
new_x = clamp(subject_field(subject, "x"), sx - max_x_offset, sx + max_x_offset)
new_y = clamp(subject_field(subject, "y"), sy - max_y_offset, sy + max_y_offset)
```

### 3. 对象一一对应

`reconcile_scene_to_manifest()` 会让 pkl 的 `subjects_data` 与 object manifest 对齐：

```python
expected_counts = Counter(obj["type"] for obj in manifest["objects"])
actual_counts = Counter(subject["type"] for subject in subjects(scene_dict))
```

缺失对象会被创建，多余对象会被删除，并写入 repair actions。

### 4. 自动闭环入口

`run_scene_harness.py` 的主循环是：

```python
agent_text2pkl_v5.py -> agent_opinion.py -> agent_reverse.py --repair-plan
```

每轮读取 `validation.json`，如果 `overall_pass=true` 就停止；否则读取 `repair_plan.json` 修复并继续。

## 三、新代码的全部功能

新增功能包括：

1. 根据 prompt 或 `--objects` / `--object-manifest` 生成主体清单。
2. 强制 pkl 中对象与主体清单一一对应，不能多、不能少。
3. 抽取并保存空间谓词：`centered`、`left_of`、`right_of`、`in_front_of`、`behind`、`above`、`below`、`support`。
4. 将谓词转换为 pkl 参数约束，直接修改 `dims/x/y/z/azimuth/camera_data`。
5. 约束 `dims` 到资产参考尺寸范围。
6. 检查并修复对象体积顺序和同类对象尺寸一致性。
7. 检查相机 `global_scale/lens/camera_elevation` 的硬范围。
8. 输出 `validation.json`、`repair_plan.json`、`repair_actions.jsonl`。
9. `agent_reverse.py --repair-plan` 支持无 LLM 的确定性修复。
10. `run_scene_harness.py` 支持全自动验证-修复循环。
11. 保存项目看板、失败知识库和 memory 管理策略。

### 1. 分层、有明确循环：计划 -> 执行 -> 验证 -> 失败恢复/再执行

结论：已经在 pkl/cube 层面实现。

各文件含义如下：

- `*.object_manifest.json`：prompt 中应该出现的物体主体清单。它是对象一一对应的事实源，要求 pkl 中不能多物体、不能少物体。
- `*.predicates.json`：从 prompt 中解析出的空间谓词，例如 `right_of(cabinet, refrigerator)`、`support(microwave, cabinet)`。

- `*.validation.json`：机器验证结果，包含 `overall_pass`、`issue_counts` 和按类别分组的失败项。
- `*.repair_plan.json`：可执行修复计划，供 `agent_reverse.py --repair-plan` 直接执行。
- `*.repair_actions.jsonl`：字段级修改轨迹，每一行是一条动作记录，适合长期积累和分析失败模式。

### 3. 有可自主运行的工具调用，而不是只靠 prompt / guidance

结论：已经在 pkl 参数修改层面实现。

原来“把 cube 往左一点”“放在 table 上”主要靠 `guidance.md` 告诉模型怎么改。现在这些行为被工具函数实现，不需要模型记住公式。

例如：

- 体积顺序会由工具检查和修复。

对应代码：

- [harness_param_constraints.py](/ptpool/users/qzy/code/seethrough3d2/harness_param_constraints.py:819)：`apply_predicates()` 根据谓词直接投影 pkl 字段。

- [harness_param_constraints.py](/ptpool/users/qzy/code/seethrough3d2/harness_param_constraints.py:589)：`normalize_dimensions()` 机械约束 dims。
- [harness_param_constraints.py](/ptpool/users/qzy/code/seethrough3d2/harness_param_constraints.py:602)：`enforce_volume_order()` 机械修复体积顺序。

所以现在 agent 可以调用工具修改 pkl，而不是把“怎么改”全部压在 prompt 中。

#### 3.1.2 `harness_param_constraints.py` 中的确定性参数工具

路径：

```text
/ptpool/users/qzy/code/seethrough3d2/harness_param_constraints.py
```

这个文件是本次 harness 的核心工具库。它里面的函数不是普通 prompt，而是可执行的 pkl 参数操作工具。

对象和 manifest 工具（6个）：

- `build_object_manifest()`：从 `--object-manifest`、`--objects` 或 prompt 中构建对象主体清单。
- `parse_objects_spec()`：解析 `"refrigerator:1,cabinet:1"` 这种用户明确给出的对象清单。
- `normalize_manifest()`：把 manifest 规范化成统一格式，生成稳定 `id/name/type/mention`。
- `map_asset_type()`：把文本中的物体短语映射到项目已有资产类型。
- `reconcile_scene_to_manifest()`：强制 pkl 中 `subjects_data` 与 manifest 一一对应；缺失则创建，多余则删除，名字不一致则重命名。
- `make_subject()`：缺失对象时创建一个带完整字段的默认 cube，包括 `name/type/dims/x/y/z/azimuth/bbox`。

空间几何工具（7个）：

- `dims()`：读取 `[Width, Depth, Height]`。
- `edges()`：根据 `dims/x/y/z` 计算 `left/right/front/back/bottom/top`。
- `margin_x()`：计算前后关系需要的安全间隔。
- `margin_y()`：计算左右关系需要的安全间隔。
- `subject_field()`：读取 `x/y/z/azimuth` 等 list 字段中的第一个数。
- `set_subject_field()`：把字段写回 pkl 的 list 形式，例如 `x: [value]`。
- `set_dims()`：写回 `dims`，并保证维度为正数。

谓词抽取工具（4个）：

- `extract_predicates()`：从文本中抽取 `centered/left_of/right_of/in_front_of/behind/above/below/support`。
- `split_clauses()`：把长 prompt 切成子句，减少跨句关系误抽取。
- `relation_between()`：判断一个子句中两个对象之间是否存在某类关系。
- `text_mentions_alias()`、`alias_position()`、`object_aliases()`：负责对象别名匹配和对象在子句中的位置判断。

谓词投影工具（4个）：

- `apply_predicates()`：把声明式谓词转换成 pkl 字段。例如 `right_of(A,B)` 转换成 `A.y` 的下界约束，`in_front_of(A,B)` 转换成 `A.x` 的下界约束。
- `clamp_subject_within_support_raw()`：实现 `support(A,B)`，保证 A 的 footprint 在 B 的 footprint 内，并让 `bottom(A)=top(B)`。
- `set_field_action()`：写入 `x/y/z/azimuth` 并记录 action。
- `set_dim_action()`：写入 `dims[i]` 并记录 action。

尺寸和体积工具（4个）：

- `dim_bounds()`：根据资产参考尺寸和文本中的 small/large 等修饰词计算 `dims` 上下界。
- `normalize_dimensions()`：把每个 cube 的 `dims` clamp 到允许范围。
- `volume_for_dims()`：计算体积 `Width * Depth * Height`。
- `enforce_volume_order()`：保证参考体积明显更大的物体，在当前 pkl 中也不能比小物体更小；同时约束同类物体尺寸不能严重失真。

相机和基础物理工具（2个）：

- `apply_camera_ranges()`：约束 `camera_data.global_scale`、`camera_data.lens`、`camera_data.camera_elevation` 到合理范围。
- `enforce_ground_and_collision()`：当前主要实现 `z >= 0` 的地面约束；复杂碰撞还没有完全展开。

验证工具（6个）：

- `validate_objects()`：检查对象是否和 manifest 一一对应。
- `validate_dimensions()`：检查 `dims` 是否在范围内。
- `validate_volume()`：检查体积相对顺序是否合理。
- `validate_predicates()`：检查所有空间谓词是否满足。
- `validate_camera()`：检查相机参数是否在范围内。
- `validate_scene()`：汇总以上所有验证结果，输出 `overall_pass`、`issue_counts`、`criteria`、`issues`。

修复计划工具（5个）：

- `build_repair_plan()`：把确定性修复动作转换成可执行 JSON repair plan。
- `to_repair_action()`：把内部 action 转成统一 repair action。
- `set_repair_param()`：执行单个 `set_param` 动作，支持 `dims[i]`、`x/y/z/azimuth` 和 `camera_data`。
- `apply_repair_plan()`：执行完整 repair plan，支持 `set_param`、`create_object`、`delete_object`。
- `run_harness_on_scene()`：一站式入口，完成 manifest、predicates、约束投影、验证和 repair plan 生成。

### 4.1. 对于 harness 特征 4的补充说明：什么情况下调用什么工具，什么情况下不调用什么工具

当前系统的工具调用分为“顶层脚本调用策略”和“harness 内部工具调用策略”。

#### 4.1.1 顶层脚本调用策略

第一步，一定先调用 `agent_text2pkl_v5.py`，除非未来扩展成“从已有 pkl 开始修复”的模式。

调用条件：

- 用户给了 `--scene-text`。
- 需要从文字生成初始 pkl。

不调用条件：

- 如果未来已有一个待修复 pkl，并且只想验证/修复它，则可以跳过 text2pkl，直接进入 `agent_opinion.py`。当前 `run_scene_harness.py` 还没有暴露这个入口。

第二步，每一轮都调用 `agent_opinion.py`。

调用条件：

- 已经存在当前轮的 pkl。
- 需要判断这个 pkl 是否满足对象、尺寸、体积、空间谓词和相机约束。

不调用条件：

- 没有 pkl 时不能调用。
- 如果只是想单独生成初始 pkl，不跑闭环，可以不调用它，直接使用 `agent_text2pkl_v5.py` 输出的 sidecar validation。

第三步，只有验证失败时才调用 `agent_reverse.py --repair-plan`。

调用条件：

- `validation.json` 中 `overall_pass=false`。
- 当前轮不是最后一轮。
- `agent_opinion.py` 已经写出 `repair_plan.json`。

不调用条件：

- `overall_pass=true`，说明 pkl 已满足机器约束，此时不再修复，避免过度修改。
- 已达到 `--max-rounds`，此时停止并写 `summary.json`。
- 没有 `repair_plan.json`，说明没有可执行修复计划，不应让 reverse 猜测修复。

对应代码：

```python
final_validation = load_json(validation_path)
if final_validation.get("overall_pass"):
    output_path.write_bytes(current_pkl.read_bytes())
    break
if round_idx == args.max_rounds - 1:
    break
run_command(reverse_cmd, run_dir / "commands.log")
```

第四步，只有 pkl 验证通过且传入 `--placeholder-prompt` 时才调用 `infer2.py`。

调用条件：

- `summary["overall_pass"] == true`。
- 用户传了 `--placeholder-prompt`。
- `--placeholder-prompt` 中包含 `PLACEHOLDER`，因为 `infer2.py` 需要用它替换对象占位符。

不调用条件：

- harness 失败时不调用 `infer2.py`，因为这时 pkl 布局还不可信。
- 用户没有传 `--placeholder-prompt` 时不调用，表示只需要生成/验证 pkl。
- prompt 中没有 `PLACEHOLDER` 时，`infer2.py` 会报错。

对应代码：


- 需要检查 pkl 的几何布局是否直观合理。

不调用条件：

#### 4.1.2 harness 内部工具调用策略

`build_object_manifest()` 的调用策略：

- 如果用户提供 `--object-manifest`，优先读取这个 JSON，因为这是最明确的主体清单。
- 如果没有 `--object-manifest` 但提供了 `--objects`，使用 `--objects` 构建 manifest。
- 如果两者都没有，才尝试从 prompt 自动解析。
- 如果 prompt 解析不到稳定主体，且 `--on-ambiguous-object fail/ask`，应失败并要求用户明确对象；如果是 `best_effort`，则继续但会写 warning。

`reconcile_scene_to_manifest()` 的调用策略：

- 每次生成 pkl 后都调用，因为对象一一对应是硬约束。
- 当 pkl 缺少 manifest 中的对象时，创建对象。
- 当 pkl 多出 manifest 中没有的对象时，删除对象。
- 当对象类型对但名称不稳定时，重命名为 manifest 中的稳定名称。
- 不应在没有 manifest 的情况下盲目增删物体，因为那样会破坏 prompt 主体约束。

`extract_predicates()` 的调用策略：

- 有 scene text 时调用，从文本中抽取空间谓词。
- 如果用户已经提供更强的 predicate 文件，未来可以优先使用 predicate 文件；当前主要从 prompt 和 manifest 抽取。
- 当一句话中没有同时出现两个对象和关系词时，不生成二元谓词。
- 当文本关系含糊时，不强行生成高风险谓词；此时只保留对象清单和尺寸/相机约束。

`apply_predicates()` 的调用策略：

- 只有 `predicates.json` 中存在谓词时才对空间位置做机械投影。
- `centered(A)` 调整 `A.x` 和 `A.y` 到默认中心。
- `left_of/right_of(A,B)` 调整 `A.y`，并将 `A.x` 与 `B.x` 对齐。
- `in_front_of/behind(A,B)` 调整 `A.x`，并将 `A.y` 与 `B.y` 对齐。
- `above/below(A,B)` 调整 `A.z`，并在可行时对齐 `x/y`。
- `support(A,B)` 调整 `A.x/A.y/A.z`，必要时缩小 `A.dims[0]` 或 `A.dims[1]`，保证 A 放在 B 的 footprint 内。
- 如果某个谓词引用的对象在 pkl 中不存在，不直接猜测位置，而是先由 `reconcile_scene_to_manifest()` 补齐对象。

`normalize_dimensions()` 的调用策略：

- 生成 pkl 后调用一次。
- 空间谓词投影后再调用一次，因为 support 可能会缩小 width/depth。
- 如果某个维度超出资产参考范围，则 clamp 到允许区间。
- 如果 prompt 明确包含 small/tiny/large/big 等修饰词，则使用更窄或偏大的范围。
- 不用 `camera_data.global_scale` 来修复单个物体大小失真，因为 global_scale 是全局缩放，不是单物体尺寸工具。

`enforce_volume_order()` 的调用策略：

- 在 `normalize_dimensions()` 内部调用。
- 当参考体积明显更大的物体在当前 pkl 中没有保持更大体积时，恢复到参考尺寸。
- 当同类物体体积差距超过阈值时，恢复同类参考尺寸，避免同类 cube 一个特别大、一个特别小。
- 如果两个物体参考体积接近，不强行规定谁必须大，避免引入错误偏置。

`apply_camera_ranges()` 的调用策略：

- 每轮 harness 都调用，确保相机参数不出界。
- 当 `global_scale` 超出 `[0.4, 2.5]` 时 clamp。
- 当 `lens` 超出 `[18, 80]` 时 clamp。
- 当 `camera_elevation` 超出 `[5°, 45°]` 时 clamp。
- 不用相机工具修复对象相对位置；对象空间关系必须由 `x/y/z/dims` 修复。

`validate_scene()` 的调用策略：

- 每轮修复前后都可以调用。
- `agent_opinion.py` 用它生成正式 `validation.json`。
- 如果 `issues` 为空，`overall_pass=true`，闭环停止。
- 如果 `issues` 非空，`overall_pass=false`，继续生成 repair plan。

`build_repair_plan()` 和 `apply_repair_plan()` 的调用策略：

- `build_repair_plan()` 在发现可修复动作时生成 JSON。
- `apply_repair_plan()` 只执行 JSON 中明确列出的动作。
- 不让 `agent_reverse.py` 在 `--repair-plan` 模式下自行理解自然语言 guidance。
- 如果 repair plan 为空但 validation 仍失败，说明缺少某类工具能力，应扩展 harness 工具，而不是让模型 try harder。

### 5.1. 对于 harness 特征 5的补充说明：规则和约束约束了哪些方面，是工具还是普通函数，代码路径在哪里

当前规则和约束主要覆盖 6 个方面：对象一一对应、维度范围、体积关系、空间谓词、支撑关系、相机范围。它们在代码形态上是“普通 Python 函数”，但在系统角色上是“可被 agent 调用的确定性工具”。也就是说，它们不是 prompt 规则，而是脚本会执行、会检查、会生成 repair plan 的工具函数。


约束 1：对象一一对应。

规则：

- prompt 或 `--objects` 中声明的每个物体，pkl 中必须存在。
- pkl 中不能出现 prompt 没声明的额外物体。
- 类型数量必须一致，例如 `refrigerator:1,cabinet:1,microwave:1,bowl:1` 就必须正好有 4 个 subject。
- 无法映射到资产类型的对象会被标记为 `unmapped_object`。

实现函数：

```text

normalize_manifest()

validate_objects()
```

验证类别：

```text
object_exact_match
object_type_count
unmapped_object
```

约束 2：`dims` 维度范围。

规则：

- `dims[0]` 是 Width。
- `dims[1]` 是 Depth。
- `dims[2]` 是 Height。
- 每个对象的 `dims` 必须落在资产参考尺寸的合理范围内。
- 普通对象范围约为参考尺寸的 `[0.70, 1.35]`。
- small/tiny/little/mini 对象范围约为参考尺寸的 `[0.45, 0.90]`。
- large/big/huge/giant 对象范围约为参考尺寸的 `[1.05, 1.80]`。

实现函数：

```text
dim_bounds()
normalize_dimensions()
validate_dimensions()
```

验证类别：

```text
dimension_range
```

约束 3：体积相对关系。

规则：

- 体积计算公式是 `Width * Depth * Height`。
- 如果资产参考体积 A 明显大于 B，那么当前 pkl 中 A 也不能比 B 小。
- 如果同类对象体积差距过大，则恢复到同类参考尺寸，避免同类 cube 一个巨大、一个极小。

实现函数：

```text
volume_for_dims()
enforce_volume_order()
validate_volume()
```

验证类别：

```text
volume_order
same_type_consistency
```

约束 4：空间相对关系。

当前支持的谓词：

```text
centered(A)
left_of(A, B)
right_of(A, B)
in_front_of(A, B)
behind(A, B)
above(A, B)
below(A, B)
support(A, B)
```

坐标语义：

- `x` 控制前后，数字变大是向前，数字变小是向后。
- `y` 控制左右，数字变大是向右，数字变小是向左。
- `z` 控制上下，数字变大是向上，数字变小是向下。

几何边界由 `edges()` 计算：

```text
left   = y - width / 2
right  = y + width / 2
back   = x - depth / 2
front  = x + depth / 2
bottom = z
top    = z + height
```

具体规则：

- `centered(A)`：要求 `A.x` 接近默认中心 `-6.0`，`A.y` 接近 `0.0`。
- `left_of(A,B)`：要求 `right(A) <= left(B) - margin_y`。
- `right_of(A,B)`：要求 `left(A) >= right(B) + margin_y`。
- `in_front_of(A,B)`：要求 `back(A) >= front(B) + margin_x`。
- `behind(A,B)`：要求 `front(A) <= back(B) - margin_x`。
- `above(A,B)`：要求 `bottom(A) >= top(B) + margin_z`。
- `below(A,B)`：要求 `top(A) <= bottom(B) - margin_z`。
- `support(A,B)`：要求 `bottom(A)=top(B)`，并且 A 的 footprint 完全落在 B 的 footprint 内。

实现函数：

```text
extract_predicates()
apply_predicates()
clamp_subject_within_support_raw()
validate_predicates()
edges()
margin_x()
margin_y()
```

验证类别：

```text
spatial_relation
```

约束 5：支撑关系和贴合关系。

规则：

- 文本中出现 `on`、`on top of`、`placed on`、`sits on`、`countertop` 等支撑提示时，优先解析为 `support(A,B)`。
- `support(A,B)` 不只是要求 A 的 `z` 大于 B，而是要求 A 的底面贴住 B 的顶面。
- 如果 A 的宽度或深度大于 B 的支撑面，工具会先缩小 A 的 `dims[0]` 或 `dims[1]`，再把 A clamp 到 B 的 footprint 内。

实现函数：

```text
extract_predicates()
clamp_subject_within_support_raw()
apply_predicates()
validate_predicates()
```

对应 pkl 字段：

```text
A.dims[0]
A.dims[1]
A.x
A.y
A.z
```

约束 6：相机范围。

规则：

- `camera_data.global_scale` 控制所有物体整体缩放，允许范围 `[0.4, 2.5]`。
- `camera_data.lens` 控制焦距，允许范围 `[18, 80]`。
- `camera_data.camera_elevation` 控制俯仰角，允许范围 `[5°, 45°]`，代码中以 radian 存储和检查。

实现函数：

```text

validate_camera()
```

验证类别：

```text
camera_range
```

约束 7：基础地面约束。

规则：

- 对象的 `z` 不应小于 0。
- 如果对象落到地面以下，工具会把 `z` 修正到 0。

实现函数：

```text

```

当前边界：

- 复杂碰撞检测还没有完整实现。
- 当前主要保证地面约束和 support 几何约束。
- 如果未来要验证“桌子不能穿墙”“物体不能互相穿模”，应继续扩展 `enforce_ground_and_collision()` 和对应 `validate_collision()`。

## 2026-05-13 追加修正：复合关系与反馈修复闭环

### 问题

在 sofa/cat/dog/lamp/bookshelf 测试中暴露出四个问题：

1. prompt 中出现 `front-right`、`front-left` 这类复合短语时，parser 能拆成两个 predicate，但执行阶段会互相覆盖。例如 `front-right(cat, sofa)` 被拆成 `in_front_of(cat, sofa)` 和 `right_of(cat, sofa)` 后，`in_front_of` 先把 `cat.x` 放到 sofa 前方，随后 `right_of` 又把 `cat.x` 对齐回 sofa，导致“前方”关系失效。
2. `agent_opinion.py` 打印出的自然语言修改意见没有进入 `repair_plan.json`。例如提示 `lamp 的所有维度减小`，但 `agent_reverse.py --repair-plan` 只执行 JSON plan，不执行自然语言 guidance，所以 lamp 实际没有变小。
3. A-F 关系检查存在误匹配。`cat` 会被正则误匹配到 `located` 这个单词内部，导致 prompt 里 `bookshelf is located to the right of the sofa` 被错误判成 `bookshelf|cat` 右侧关系。
4. 多个 cube 出画时，旧 harness 只报告 `frame_clipping`，没有把该问题转成相机或尺寸参数修改。

### 修改

#### 1. 复合关系联合执行

修改 `harness_param_constraints.py::apply_predicates()`：

- 先统计同一 `(subject, object)` 对应的全部 predicate 类型。
- 如果同一对物体同时有前后关系和左右关系，则：
  - `left_of/right_of` 只修改 `y`，不再强制对齐 `x`。
  - `in_front_of/behind` 只修改 `x`，不再强制对齐 `y`。

这样下面这些复合短语都能同时满足两个方向：

```text
front-right -> in_front_of + right_of
front-left  -> in_front_of + left_of
back-right  -> behind + right_of
back-left   -> behind + left_of
```

#### 2. A-F 反馈转为可执行 repair plan

新增 `harness_param_constraints.py::apply_feedback_issues()`、`build_feedback_repair_plan()`、`merge_repair_plans()`：

- `agent_opinion.py` 仍负责生成 A-F 评估与中文/英文建议。
- 但现在会额外把 A-F issue 编译成字段级 repair action。
- 最终写出的 `repair_plan.json` 是 `harness predicate repair plan + feedback repair plan` 的合并结果。

典型映射：

```text
frame_clipping(少数 cube) -> target.dims *= 0.92
frame_clipping(多个 cube) -> camera.lens *= 0.80, camera.global_scale *= 0.90
relation_right_mismatch(A|B) -> A.y = B.y + B.width/2 + A.width/2 + margin
relation_front_mismatch(A|B) -> A.x = B.x + B.depth/2 + A.depth/2 + margin
below_ground/floating_object -> z = 0
size_implausible -> dims = visual_target_dims(type)
```

这样 `run_scene_harness.py -> agent_opinion.py -> agent_reverse.py --repair-plan` 的链路不再依赖 prompt 或自然语言 guidance 执行。

#### 3. 修复关系匹配假阳性

修改 `agent_check_pkl_v5.py::ordered_relation_in_clause()` 和 `agent_reverse_core.py::ordered_relation_in_clause()`：

- 对英文 alias 增加词边界匹配。
- 先确认 second alias 真的作为对象名出现，再构造 relation 正则。

修复后，`cat` 不会再匹配到 `located` 内部，`bookshelf is located to the right of the sofa` 不会被误判为 `bookshelf` 在 `cat` 右侧。

#### 4. 有 centered 谓词时不再整体平移破坏中心对象

`center_scene_component()` 原先会在所有谓词投影后整体平移组件，有时会把已经满足 `centered(sofa)` 的 sofa 移走。

现在 `apply_full_harness()` 会收集 `centered` predicate 对应对象；如果存在 pinned center object，则跳过整体组件居中，避免 harness 自己破坏中心约束。

#### 5. 新增问题到解决方案文档

新增：

```text
docs/harness/FEEDBACK_REPAIR_PLAYBOOK.md
```

该文档把评估问题流程化映射到机械修复动作，说明每类 issue 应该修改哪些 pkl 字段。例如：

- 单个 cube 出画：缩小该 cube 的所有维度。
- 多个 cube 出画：降低焦距和整体 scale。
- 左右/前后关系错误：按边界和 margin 直接重算 `x/y`。
- 地面/漂浮问题：直接修正 `z`。

## 2026-05-14 追加修正：repair_plan 执行一致性与空转检测

### 问题

`runs/20260513_225932_scene_harness` 暴露出一个 repair plan 格式与执行器不一致的问题：


```text

```

### 修改

#### 1. 支持整组 `dims` repair action

修改 `harness_param_constraints.py::set_repair_param()`：

- 支持 `param == "dims"`。
- 要求 `value` 必须是长度为 3 的 list/tuple。
- 执行时一次性写入 `[width, depth, height]`。
- 如果是缩小所有尺寸，要求三个维度缩放比例一致，避免“所有维度减小”被执行成非等比例变形。

```json

```

则会真实修改：

```text
lamp.dims: [0.3, 0.3, 1.2] -> [0.276, 0.276, 1.104]
```

#### 2. 无效 action 不再静默跳过

修改 `harness_param_constraints.py::apply_repair_plan()`：

- 如果 `set_param` action 无法执行，会抛出错误。
- 不再允许 repair plan 中有 action，但实际没有任何参数变化。

这保证“修改意见没有落实”的问题会在执行阶段暴露，而不是继续进入下一轮。

#### 3. `run_scene_harness.py` 增加空转检测

新增检查：

```text

applied_actions = repair_actions.jsonl 的实际行数
same_pkl = 当前 pkl bytes == 修复后 pkl bytes
```

```text
planned_actions > 0 and (applied_actions == 0 or same_pkl)
```


## 十二、屏幕空间前后分级约束设计

### 1. 问题

`front-right/front-left/back-right/back-left` 的世界坐标前后边缘距离已经加大到 `1.5`，但在实际生成图像中，前景对象仍可能没有明显进入画面下半部分。例如 cat/dog 虽然已经在 sofa 前方，但投影到图像后仍离图片底部较远，导致画面前半部分显得空。

因此，仅靠世界坐标中的 A-B 距离不够，需要增加一层屏幕空间约束，直接控制 A 投影后距离图片底部或顶部的像素距离。

### 2. 文档规则

已更新：

```text
docs/harness/PROMPT_RELATION_RULES.md
```

新增 `Screen-Space Front/Back Depth Levels` 小节，规定：

- 使用投影后的 cube bbox 像素坐标，不使用世界坐标距离。
- 前方关系以图片底部为基准：

```text
bottom_gap(A) = image_height - bbox_bottom(A)
bottom_gap(A) <= bottom_gap(B) - delta_px
```

- 后方关系以图片顶部为基准：

```text
top_gap(A) = bbox_top(A)
top_gap(A) <= top_gap(B) - delta_px
```

- 复合短语必须先同时满足空间 predicate。例如 `front-right(A, B)` 必须先满足 `in_front_of(A, B)` 和 `right_of(A, B)`；`back-left(A, B)` 必须先满足 `behind(A, B)` 和 `left_of(A, B)`。
- 单纯 `in front of` / `behind` 也必须先满足世界坐标前后关系，再执行屏幕空间分级。
- 保持参照物 `B` 不动，只移动 `A` 的前后轴位置来接近屏幕目标。
- 如果屏幕目标与 `full_cube_visible` 冲突，完整可见性优先。

### 3. 分级表

前方 / 右前 / 左前：

```text
slightly:    bottom_gap(A) <= bottom_gap(B) - 80px
default:     bottom_gap(A) <= bottom_gap(B) - 160px
clearly:     bottom_gap(A) <= bottom_gap(B) - 260px
far:         bottom_gap(A) <= bottom_gap(B) - 340px
extra-far:   bottom_gap(A) <= bottom_gap(B) - 420px
```

后方 / 右后 / 左后：

```text
slightly:    top_gap(A) <= top_gap(B) - 80px
default:     top_gap(A) <= top_gap(B) - 160px
clearly:     top_gap(A) <= top_gap(B) - 260px
far:         top_gap(A) <= top_gap(B) - 340px
extra-far:   top_gap(A) <= top_gap(B) - 420px
```

安全留白建议：

```text
min_visible_bottom_gap_px = 80
min_visible_top_gap_px = 80
```

### 4. 后续代码实现口径

该规则必须通过 harness 的投影函数和数值移动函数机械执行，而不是依赖 LLM prompt。后续实现时应复用已有投影能力：

```text

project_point_to_image()
projected_component_bbox()
```

建议新增单物体 bbox 投影函数，计算每个 subject 的 `bbox_top/bbox_bottom`，然后对前方关系通过增大 `A.x` 让 `A` 更靠近图片底部，对后方关系通过减小 `A.x` 让 `A` 更靠近图片顶部。每次移动后必须重新验证 predicate 和 `full_cube_visible`。

## 十五、screen-size 屏幕尺寸合理性约束

### 1. 问题

前后关系会把 `cat/dog` 推到更靠近镜头的位置。即使它们的世界坐标 `dims` 等于 `inference/asset_dimensions.json`，投影后也可能因为透视显得比 `sofa/lamp` 更大，造成“远近正确但屏幕大小不合理”。

### 2. 规则

新增 screen-size 校验，使用投影 bbox 像素计算：

```text
bbox_width  = projected_bbox.max_x - projected_bbox.min_x
bbox_height = projected_bbox.max_y - projected_bbox.min_y
bbox_area   = bbox_width * bbox_height

height_density = bbox_height / asset_height
area_density   = sqrt(bbox_area / (asset_width * asset_height))
density        = 0.75 * height_density + 0.25 * area_density

screen_size_ratio = density / median(scene_density)
```

强制范围：

```text
0.8 <= screen_size_ratio <= 1.2
```

### 3. 代码修改

修改 `harness_param_constraints.py`：

- 新增 `_screen_size_records()` / `_screen_size_target_density()` / `_screen_size_ratio()`，用投影 bbox 与 `asset_dimensions.json` 参考尺寸计算屏幕尺寸比例。
- 新增 `enforce_screen_size_reasonableness()`，在 `apply_full_harness()` 中执行。
- 修复顺序固定为：先调目标 cube 的 `x`，再尝试 `camera.lens/global_scale`，必要时才等比例补偿 `dims`。
- 带 `front/behind` screen-depth 的对象，screen-size 调 `x` 时不能破坏 `bottom_gap/top_gap` 分级。
- 新增 `_screen_size_dim_compensated` 标记。若必须补偿 `dims`，repair plan 会写入该标记，后续轮次不会被尺寸表重新拉回。
- 新增 `_restore_screen_depth_after_screen_size()`，在尺寸补偿后再次恢复前/后屏幕分级。
- `validate_scene()` 增加 `screen_size_reasonableness` criterion。
- `dimension_range` / `volume_order` 对 `_screen_size_dim_compensated=true` 的对象跳过，避免和 screen-size 约束冲突。

修改 `agent_check_pkl_v5.py`：

- `summarize_scene_subjects()` 保留 `_screen_size_dim_compensated` 标记。
- 本地 A-F 尺寸检查对该标记对象跳过 `size_implausible/size_suspicious/size_order_conflict`。

修改 `agent_opinion.py`：

## 十六、screen-size 前后关系分级范围与 dims 安全边界

### 1. 问题

上一版 screen-size 约束对所有物体统一使用：

```text
0.8 <= screen_size_ratio <= 1.2
```

这会把 `front-right/front-left/front` 的前景物体强行拉回和背景物体接近的屏幕密度。以 sofa/cat/dog/lamp/bookshelf 场景为例，cat/dog 被 screen-depth 推到 sofa 前方后，初始投影 ratio 明显高于 1.2，随后触发多轮 `screen_size:dim_compensation`，导致 cat/dog world `dims` 被压到低于默认尺寸 0.7x。

### 2. 新规则

screen-size 允许范围改为由 prompt 谓词机械决定：

```text
普通物体:                          [0.8, 1.2]
front / front-left / front-right:  [0.8, 1.6]
clearly front:                     [0.75, 1.8]
far / very far / extra-far front:  [0.7, 2.0]
back / back-left / back-right:     [0.55, 1.2]
clearly back:                      [0.45, 1.15]
far / very far / extra-far back:   [0.35, 1.1]
```

同时新增 per-asset `dims` 安全边界：

```text
min_dims = asset_dimensions[type] * 0.7
max_dims = asset_dimensions[type] * 1.4
```

该边界适用于 screen-size compensation 和 repair plan 写入，确保 cat/dog 等小物体不会被缩到全局 `MIN_VISUAL_DIMS`。

### 3. 代码修改

- `harness_param_constraints.py`
  - 新增 `SCREEN_SIZE_FRONT_RATIO_RANGE_BY_LEVEL` / `SCREEN_SIZE_BACK_RATIO_RANGE_BY_LEVEL`，按 `default/clear/far/extra_far` 分级选择 screen-size 允许范围。
  - 新增 `DIM_REFERENCE_MIN_RATIO = 0.7`、`DIM_REFERENCE_MAX_RATIO = 1.4`。
  - 新增 `_screen_size_ranges_by_subject()`，从 `in_front_of/behind` predicate 读取 subject 的 front/back 分级并返回 per-subject ratio range。
  - `enforce_screen_size_reasonableness()`、`_try_camera_for_screen_size_reasonableness()`、`_restore_screen_depth_after_screen_size()` 和 `validate_screen_size_reasonableness()` 不再使用固定 `[0.8, 1.2]`，改用 per-subject range。
  - `_scale_subject_dims_uniform()` 对 screen-size 补偿结果做 `[0.7x, 1.4x]` clamp。
  - 新增 `_screen_size_expand_reference_subjects()`：当前景对象仍超过 ratio 上限、且继续缩小会触碰 0.7x 下限时，优先把不参与空间谓词边界的中性参考物体在 1.4x 上限内放大，提高 median density，避免继续压缩 cat/dog；被放大的参考物体写入 `_screen_size_dim_compensated=true`。
  - `validate_screen_size_reasonableness()` 增加安全边界优先级：对象已到 0.7x 下限时，不再为了满足 front ratio 继续要求缩小；对象已到 1.4x 上限时，不再为了满足 back ratio 继续要求放大。
  - `enforce_screen_size_reasonableness()` 禁止对无 front/back screen-depth、但受 left/right 普通谓词约束的对象单独调 `x`，避免与 `left_of/right_of` 的 `x` 对齐动作来回抵消；这类对象仍可通过相机或 `[0.7x, 1.4x]` 内的 dims 补偿修复 screen-size。
  - 新增 `SCREEN_DEPTH_TOLERANCE_PX = 3.0`，screen-depth 执行和验证统一使用 3px 容差，避免 1-2px 投影误差导致 repair loop 震荡。
  - `apply_full_harness()` 在 screen-size / screen-depth 修复后再次执行空间谓词、支撑展开和组件居中，避免参考物体放大后破坏 `left/right/front/back` 边界。
  - `validate_dimensions()` 对 `_screen_size_dim_compensated=true` 对象不再完全跳过，而是检查 `[0.7x, 1.4x]` 安全边界。
  - `set_repair_param()` 对 repair plan 中的 `dims` / `dims[i]` 写入也执行 `[0.7x, 1.4x]` clamp。
- `agent_check_pkl_v5.py`
  - 本地 D_size_reasonable 的 hard size 上限从 1.35 同步为 1.4，和 harness 的 `max_dims = asset_dimensions[type] * 1.4` 保持一致。
  - 本地 visibility 检查补充 `camera_data.global_scale`，避免 A-F 的 `E_full_cube_visible` 与 harness projection 使用不同尺寸尺度。

### 4. 文档更新

- `docs/harness/LAYOUT_READABLE_DIMENSION_RULES.md`：更新 screen-size 约束为前景/背景分级范围，并写入 `dims` 默认尺寸 `[0.7x, 1.4x]` 安全边界。
- `docs/harness/FEEDBACK_REPAIR_PLAYBOOK.md`：更新 `screen_size_reasonableness` 的允许范围和 dim compensation 边界。

## 十七、最终组件构图紧凑约束

### 1. 问题

sofa/cat/dog/lamp/bookshelf 场景中，`front-left/front-right`、full-visible 和 screen-size 约束都能通过，但为了让所有 cube 进入画面，相机会被拉远，导致所有 cube 的 union bbox 只占画面很小一部分。以 `runs/20260517_194652_scene_harness/round_03_scene.pkl` 为例：

```text
component_width_ratio  ~= 0.67
component_height_ratio ~= 0.21
```

layout 关系已经合理，但主体周围留白过多，sofa 看起来偏小。

### 2. 新规则

新增 final camera compactness 机械约束。对所有 cube 的投影 bbox 取 union bbox，分别计算：

```text
component_width_ratio  = union_bbox_width  / image_width
component_height_ratio = union_bbox_height / image_height
```

不设置显式上限。规则是：先满足最低宽高占比，然后在 full-visible、spatial relation、screen-depth、screen-size 和 camera range 都不失败的前提下，尽可能增大主体。上限由安全边距和硬规则自然决定。

最低比例按场景类型机械选择：

```text
单物体:                              width >= 0.35, height >= 0.45
2-3 个物体，普通关系:                 width >= 0.55, height >= 0.38
通用 fallback:                        width >= 0.65, height >= 0.32
4-6 个物体，横向展开:                 width >= 0.70, height >= 0.30
4-6 个物体，有 front/back 层次:       width >= 0.65, height >= 0.34
7 个以上物体:                         width >= 0.78, height >= 0.28
明显纵向物体为主:                     width >= 0.45, height >= 0.50
```

如果同时满足多种场景类型，取更严格的宽高最低值。例如当前 5 物体场景同时包含 `left/right` 和 `front-left/front-right`，目标为：

```text
width >= 0.70
height >= 0.34
```

安全边距：

```text
COMPONENT_COMPACT_MARGIN_PX = 32
COMPONENT_COMPACT_MIN_MARGIN_PX = 20
```

### 3. 代码修改

- `harness_param_constraints.py`
  - 新增 `COMPONENT_COMPACT_*` 参数。
  - 新增 `component_bbox_metrics()`，计算 union bbox 的宽高比例、面积比例和四周边距。
  - 新增 `component_compactness_target()`，根据 object count、`left/right`、`front/back` 和纵向物体比例机械选择最低宽高目标。
  - 新增 `validate_component_compactness()`，把构图过松作为 harness issue 输出。
  - 新增 `enforce_component_compactness()`，通过相机搜索收紧画面；优先增大 `camera.lens`，必要时增大 `camera.global_scale`，不优先移动 cube，不优先改 dims。
  - `apply_full_harness()` 在 screen-size 和 camera range 后执行 component compactness，并再次执行 screen-depth/screen-size，避免相机收紧后破坏前后关系。
  - `validate_scene()` 新增 `component_compactness` criterion。
  - `SCREEN_SIZE_RATIO_TOLERANCE = 0.01`，避免 `1.600009 > 1.60` 这类浮点边界误差导致 repair loop 卡死。
- `agent_opinion.py`
  - 新增 `component_compactness` 的中文/英文建议，说明应使用 harness component-compactness 工具收紧相机。
- `run_scene_harness.py`
  - 如果最后一轮失败但 repair plan 有 action，会额外执行一次 final repair 并重新验证；通过后仍会复制到用户指定 `--output`。这避免 `round_03` 已经生成有效 repair plan 但 `--max-rounds 4` 没机会落地。

## 十七、左右屏幕硬间距、pairwise 遮挡上限与相机紧凑化收敛修复

### 1. 问题

新增 component-compactness 后，`inference/saved_scenes/harness_test5.4_6.jpg` 的主体占比可以变大，但 `lamp` 和 `sofa` 曾经出现几乎完全遮挡。根因是：

- 世界空间 `left_of/right_of` 只保证边界顺序，没有要求投影后的屏幕像素间隔。
- component-compactness 拉近相机后，原本可接受的左右世界间距可能在屏幕上变成明显重叠。
- 最后一轮 repair 过去只额外执行一次，遇到 `screen-depth -> component-compactness -> screen-size/screen-lateral` 这种多步收敛时会提前停止。

### 2. 新增机械约束

在 `harness_param_constraints.py` 中新增：

```text
SCREEN_LATERAL_GAP_PX_BY_LEVEL = {
    slight: 24,
    default: 48,
    clear: 80,
    far: 120,
    extra_far: 160,
}

PAIRWISE_OCCLUSION_MAX_RATIO = 0.20
```

所有 `left_of/right_of` predicate 都会生成 `screen_lateral_axis/screen_lateral_level/screen_lateral_gap_px`，并由 `enforce_screen_lateral_gaps()` / `validate_screen_lateral_gaps()` 强制执行：

```text
left_of(A, B):  bbox_right(A) <= bbox_left(B)  - gap_px
right_of(A, B): bbox_left(A)  >= bbox_right(B) + gap_px
```

任意两个 cube 还必须满足：

```text
intersection_area(bbox(A), bbox(B)) / min(area(A), area(B)) <= 0.20
```

修复函数为：

```text
enforce_pairwise_screen_occlusion()
validate_pairwise_screen_occlusion()
```

### 3. 与旧规则的兼容

没有削弱原有规则：

- 世界空间 `left_of/right_of/front/back` 仍由 `apply_predicates()` 和 `validate_predicates()` 验证。
- `front-right/front-left/back-right/back-left` 仍同时满足前后和左右两个 predicate。
- screen-depth 前后分级继续执行，默认 `default = 80px`，`clear/far/extra_far` 保持更强分级。
- screen-size ratio、dims 安全范围和 full-visible 仍必须通过。
- component-compactness 的候选相机必须同时通过 screen-depth、screen-lateral、pairwise occlusion、screen-size 和 full-visible。

### 4. 相机紧凑化修复

`enforce_component_compactness()` 已从只搜索 `lens/global_scale` 改为联合搜索：

```text
camera.lens
camera.global_scale
camera.camera_elevation
```

每个候选先检查不破坏空间关系、screen-depth、screen-lateral、pairwise occlusion、full-visible 和 camera range，再执行后验修复：

```text
screen-size -> screen-depth -> screen-lateral -> pairwise occlusion -> camera range
```

只有后验验证全部通过，候选才会被接受。这样相机收紧不会重新制造遮挡，也不会牺牲已有前后/左右关系。

### 5. run_scene_harness 收敛修复

`run_scene_harness.py` 新增最多 `FINAL_REPAIR_EXTRA_PASSES = 3` 次最终机械 repair。最后一轮失败但 repair plan 仍有可执行 action 时，会继续执行：

```text
repair -> validate -> new repair plan
```

直到通过、无 action 或检测到 stalled。这样不会因为 `--max-rounds 4` 的最后一轮刚生成有效修复计划而提前失败。
