Harness Engineering实践心得：如何高效驾驭AI？


关注腾讯云开发者，一手技术干货提前解锁👇



先做个自我介绍。我是一名游戏客户端开发工程师，日常工作在 Unity 引擎开发。从去年开始高强度使用 AI 辅助开发，一开始只是让它帮我补补代码、查查 API，后来越用越深入，逐渐突破了自己原有的技术边界——借助 AI 的能力和公司内网提供的工具链，我独立给项目组交付了 WPF 桌面启动器、好几个内部提效的 Web 站点、还有一堆企业微信机器人。这些东西放在以前，对一个纯客户端出身的人来说几乎不可能独自完成。正是这段经历让我对"如何高效驾驭 AI"这件事有了很多切身体会，也是写这篇文章的出发点。





01


从 Prompt 到 Context 再到 Harness：AI 编程的三次跃迁


   1.1 什么是 Harness Engineering？



2026 年 2 月，OpenAI 工程师 Ryan Lopopolo 发了一篇很有分量的文章——Harness engineering: leveraging Codex in an agent-first world。里面披露了一组数据：一个小团队用五个月时间让 Codex 产出了大约 100 万行代码、1500 个 PR，全程零手写代码。这不是实验室里的 demo，是有内部日活用户的真实产品。



OpenAI 管这种工作方式叫 Harness Engineering。Harness 这个词本意是"马具、挽具"——马再好，不套上鞍具它也拉不了车。给 AI Agent 套上合适的 Harness，它才能稳定地干活。



文章里有句话我反复读了好几遍：



“When something failed, the fix was almost never ‘try harder.’ Because the only way to make progress was to get Codex to do the work, human engineers always stepped into the task and asked: ‘what capability is missing, and how do we make it both legible and enforceable for the agent?’ ”



说白了就是：Agent 搞砸的时候，解决办法不是"再跑一次试试"，而是退一步去想"缺了什么护栏？怎么让 Agent 看得懂、绕不过？"

02


一个真实案例：JK Launcher 的进化之路


   2.1 项目背景



JK Launcher 是我给项目组做的一个 Unity 项目管理工具，用 WPF (.NET Framework 4.8) 写的。它干的事情不复杂但很琐碎——管 SVN 更新、启动 Unity 引擎、替换 Library 缓存、处理冲突、跑定时任务，都是游戏开发团队每天离不开的日常操作。



这个项目从第一行代码开始就是 AI 写的。从 V3.0 到现在的 V3.11，经历了 12 个大版本迭代，是我个人实践 AI 编程方法论的试验田。



   2.2 Prompt 时代（V3.0 ~ V3.3）



最早的做法非常粗暴：



我：帮我写一个 WPF 窗口，左边项目列表，右边详情，有个"更新"按钮调 SVN update。
AI：好的，代码如下...


一问一答，每次对话都是从零开始。AI 不知道我们项目有什么规范、用了什么框架、昨天改了什么。



痛点显而易见：

每次都得把项目背景重新交代一遍

AI 经常写出风格不一致的代码——比如它习惯用 MessageBox.Show，而我们封装了 MessageDialog.Show

改一个地方不知道会不会把另一个地方搞坏

版本少的时候靠记忆还能管，版本一多就彻底失控



   2.3 Context 时代（V3.4 ~ V3.8）



图片


项目复杂了以后，我开始学着写设计文档来引导 AI。V3.5 是个转折点——我第一次正儿八经写了一份 V3.5_DESIGN_SPEC.md，将近 1000 行，把功能描述、解决方案、文件变更列表、UI 布局的 ASCII 示意图、本地化键值表、按钮配色方案、测试清单、风险评估全塞进去了。



效果立竿见影：AI 不用猜我想要什么颜色什么交互，照着文档实现就行，出活质量一下上了台阶。



但新问题跟着来了：

文档管得了"做什么"，管不了"怎么做"——AI 照样可能写出 C# 8.0 语法、忘加日志前缀、忘做本地化

文档会过时，新版本的对话窗口看不到老版本写过的约定

知识散落各处：有些在聊天记录里，有些在文档里，有些只在我脑子里



   2.4 Harness 时代（V3.9 至今）



图片


从 V3.9 开始，我的思路变了。不再只是想"怎么把 Prompt 写得更好"或者"怎么把设计文档写得更全"，而是开始搭系统：

声明式规则（.cursor/rules/）：把编码约束固化成规则文件，AI 每次启动都会加载

技能封装（.cursor/skills/）：把编译、测试之类的重复操作包装成一键可调的"技能"

自动化验证（verify_all.ps1）：改完代码后跑一遍自动检查，看有没有违规

多 Agent 协作（.cursor/agents/）：7 个角色各管各的，流水线作业

基线管理（verification_baseline.json）：量化指标只升不降，测试数量少了立刻报警



关键转变：以前是"告诉 AI 做什么"，现在是"建一个 AI 能自己转起来的环境"。这就是 Harness。





03


搭建 Harness：具体做了什么


   3.1 多 Agent 协作体系



图片


为什么不能只用一个 Agent？



很自然的疑问：一个 AI 全干了不行吗，为什么要拆成七个？



道理其实和现实中的软件工程一样——同一个人既写需求又写设计又写代码又审自己的代码，盲区一定很大。AI 也一样。我在第一版用单 Agent 做的时候踩过这些坑：



问题
具体表现
自我确认	
写代码的 Agent 不会主动推翻自己的设计，"我设计的一定没问题"
上下文过载	
一个复杂需求涉及十几个文件的改动，全塞进一个上下文里，漏改、改错的概率直线上升
没有质量门	
设计有问题也会被一路带进代码里，发现的时候已经写了几百行
缺少审计	
出了问题说不清"当时为什么这么定的"，因为所有决策都散落在一次对话里


拆成多个 Agent 以后，每个角色只管一类事，互相之间通过正式文档交接。写需求的管写需求，审方案的管审方案，写代码的不碰需求文档——职责分离本身就是一种质量保障。

光看这张表可能觉得"不就是分了个工嘛"。但每个角色存在的意义，得结合实际跑出来的结果才能体会到。我用之前做 Galileo 共享库抽取时的真实经历来说明——那个需求要从 1687 行的伽利略遥测服务中抽出一个独立 DLL，涉及三个工程的联动改动，是一次完整的七阶段实战：



Requirement Analyst 不只是把设计稿翻译成需求列表，它会真的去读源码做交叉比对。那次它发现设计稿里压根没提到一个问题：Logs 和 Metrics 用的 OTLP 属性键名不一致（一个叫 server，另一个叫 service.name）。如果没有这个阶段，这个不一致会被带进新 DLL 的 API 设计，后面要改就麻烦了。



Solution Architect 做的最关键的一个决策是：ScreenCount（走 GetSystemMetrics）可以放进 DLL，但 ScreenResolution/DPI（走 WPF 的 SystemParameters）不能，因为 DLL 不允许依赖 WPF。这种"哪些能带走、哪些必须留下"的判断，单凭一个写代码的 Agent 是做不好的——它的本能是"把所有功能都搬过去"。



Gate Reviewer 是我认为性价比最高的环节。它不信任前面 Agent 的结论，会自己打开文件验证。那次它发现方案设计的迁移清单里少列了一个文件（LauncherConfigModels.cs）——如果直接开工，V3 迁移完以后编译会直接挂。一个 Agent、一份文档，拦住了一次"设计时没想到→开发时编译不过→花两小时排查"的返工。



Developer Agent 只管按方案落地。它不做设计决策，拿着明确的 API 签名、文件列表、编码约定，按部就班地实现。给它一份清晰的规格，出活质量远好于让它"自由发挥"。



Code Reviewer 以"外人"视角看代码，抓到了 Developer 自己不会注意到的问题：ProjectGuid 重复（会导致 VS 解决方案加载异常）、GalileoSpan.Dispose 里缺 try-catch（违反 NFR-04 稳定性要求）。自己审自己的代码，这种问题永远发现不了。



QA Tester 跑了 12 项检查清单，其中最重要的一项是搜索确认没有残留的旧命名空间引用——这种"搜索型验证"人工很容易遗漏。



PM Orchestrator 全程做的是一件事：看每个阶段的结论，决定能不能往下走。闸门说"有条件通过"，它就确认条件能不能接受；评审说"1 个 Major 必改"，它就把开发退回去改完再来。到最后收拢各阶段结论，给出"可以上线"的最终判定。



每个 Agent 都抓到了前面 Agent 没抓到的东西。这就是流水线的价值——不是 7 倍工作量，而是 7 层过滤网。



模型选择的成本考量



只有 Developer Agent 用最贵的模型（claude-4.6-opus-high-thinking）。其余 6 个全用 composer-2，因为它们本质上在做文档处理而不是代码生成。Code Reviewer 用的是 gpt-5.4-medium，介于两者之间——它需要深度理解代码但不需要自己写。光模型选择这一项就省了不少 token 开销。



   3.2 PM 怎么控制流程



PM Orchestrator 是整套体系的指挥中心，它跑的是一个状态机：



[需求分析] → [方案设计] → [闸门评估] → [开发实现] → [代码评审] → [测试验证] → [交付]
                              ↓                            ↓              ↓
                         回退到需求/方案               回退到开发       回退到开发


几条硬规矩：

闸门有阻塞项 → 不许开工，必须回退

评审有必改项 → 不许进测试，必须回退

测试有阻塞缺陷 → 回退到开发修

同一阶段连续回退 3 次 → 停下来跟用户汇报，建议重审需求



每个 Agent 的输出不是聊天消息，而是落地的文档，按固定目录结构存放：



docs/features/SchedulePostUpdateBehavior/
├── 01_REQUIREMENT_ANALYSIS.md    ← 需求分析
├── 02_SOLUTION_DESIGN.md         ← 方案设计
├── 03_GATE_REVIEW.md             ← 闸门评估
├── 04_DEVELOPMENT.md             ← 开发记录
├── 05_CODE_REVIEW.md             ← 代码评审
└── 06_TEST_REPORT.md             ← 测试报告


OpenAI 文章里讲的 "Repository knowledge is the system of record"——仓库里的知识才是唯一的真相源头——我们在这里算是做到了。



   3.3 Agent 体系的迭代过程



从第一版起就是 7 个 Agent，这个骨架没动过。但光定义角色远远不够，真正费劲的是把角色之间的交互规则理清楚。



第一版的致命问题：没人管"不通过怎么办"



第一版的最大坑是：文档里没写某个环节不通过时该怎么办。



结果呢？



Gate Reviewer 审出方案有漏洞——然后它自己把需求文档改了，改完自己再审一遍，过了。标准的"既当运动员又当裁判"。

Code Reviewer 发现代码有 bug——然后它自己把代码改了，改完自己再 review 一遍，也过了。修复质量可想而知，经常改出新 bug。



问题的根子在于：每个 Agent 只知道自己的活儿，不知道"出了问题球该传给谁"。



迭代一：定义回退路由



最关键的一次改动——给 PM 加上了完整的回退规则：



闸门评估不通过 → PM 拍板 → 退回需求分析（需求问题）或方案设计（方案问题）
                           Gate Reviewer 绝不能自己改需求改方案
代码评审不通过 → PM 拍板 → 退回开发（代码问题）或方案设计（设计跑偏）
                           Code Reviewer 绝不能自己改代码
测试不通过    → PM 拍板 → 退回开发（代码缺陷）或需求分析（需求漏了）
                           QA Tester 绝不能自己修 bug


核心就一句话：发现问题的人不能自己修。 这跟现实中的软件工程一个道理——测试不该自己改代码。



迭代二：把每个 Agent 的操作流程拆细



第一版的 Agent 定义停留在"你负责做 X"的层面，没说"X 到底包括哪些动作"。导致输出质量很不稳定。



拿 Code Reviewer 举例，第一版只要求"审查代码、给合并建议"。现在要求覆盖这些维度：

新增的"需求实现 Review"解决了一个顽疾：Developer Agent 经常把主流程写完了，但需求文档里提到的某个边界场景给漏了。现在 Code Reviewer 会拿着需求文档一条一条核对，想漏都难。



其他 Agent 也做了类似细化：

Requirement Analyst：碰到有歧义的表述（比如"去掉这个功能"），必须列出至少两种解读（彻底删除 vs 改成可配置），让用户拍板

Gate Reviewer：新增了"开发过程高概率疑问点"——提前预判开发阶段会卡在哪里

Developer Agent：编译不过禁止往下走，不准带着编译错误进评审

QA Tester：不只是写用例了，还要负责维护可执行的自动化测试项目



迭代三：PM 项目级任务看板



这是一个不起眼但实际影响很大的改进。注意，这里说的不是单个任务内部的追踪——每个任务自己的流程文档天然就在管这事——而是项目级别的历史任务索引。



为什么需要这个？因为 AI 没有记忆。一个项目持续开发下来，历史任务堆了一大堆，但每次新开一轮对话，AI 什么都不知道。没有项目看板的时候，每个新需求都在从零开始分析，两个问题躲不掉：

重复造轮子：上次明明已经做过类似功能，新任务不知道，又搞了一遍

设计走偏：新功能的思路跟已有功能拧着来，架构越做越碎



举个实际的例子。我们的"定时更新"功能之前已经做完了基础流程——定时触发、SVN 更新、状态通知。现在我想在这个基础上加一个"超时中断"的能力。如果没有看板，Requirement Analyst 对"定时更新"一无所知，大概率会设计出一套跟现有实现不兼容的方案。有了看板以后，Requirement Analyst 会先搜"定时更新"相关的历史任务：



## 项目任务看板（摘录）

| 任务ID | 任务名称 | 状态 | 文档路径 |
|---|---|---|---|
| T-012 | 定时更新基础功能 | 已完成 | docs/features/ScheduleUpdate/ |
| T-015 | 定时更新后自动打开 Unity | 已完成 | docs/features/SchedulePostUpdate/ |
| T-023 | 定时更新超时中断 | 进行中 | docs/features/ScheduleTimeout/ |


找到 T-012 和 T-015，把历史文档拉出来看一遍——现有的状态机怎么设计的、之前做过什么技术决策、哪些坑踩过了——这些上下文直接输入到新需求的分析里。新功能就是在已有地基上加层楼，而不是在旁边另起一栋。



说白了，项目看板给了 AI 一种跨会话的"记忆"。每次新对话启动，PM 读一遍看板就能重建项目全貌，不用每次都问"之前做过什么来着？"



   3.4 Rules vs Skills vs Scripts：从全是 Rules 到三层分离



这套三层架构不是预先规划的，是被现实问题一步步逼出来的。



一开始：所有东西都是 Rules



项目早期没有 Scripts，也没有 Skills。编码规范、流程要求、检查清单，统统写成 Rules（.mdc 文件），全部设为 alwaysApply: true，AI 每次启动都加载。



比如：

"XAML 里禁止出现中文"——50 行 Rule，带正确和错误示例

"禁止用 MessageBox.Show"——30 行 Rule，附替代方案

"SVN 命令必须带认证参数"——90 行 Rule，各种命令模板都列全了

"测试数量只增不减"——写在测试规范的 Rule 里



加起来 14 条 always-applied 规则，1000 多行。



问题来了：任务一长 AI 就开始"忘事"



这套方案在小任务上没问题。但任务一复杂——比如一个需求改十几个文件，经过好几轮评审和修复——AI 开始频繁踩雷：

改了 15 个文件，第 13 个里面冒出来一个 MessageBox.Show

SVN 命令丢了 --username

XAML 里写了中文



Rule 里全都明明白白写着呢，但 AI 就是没照做。



原因其实不难理解——注意力衰减。1000 多行 Rules 占着上下文窗口的前排位置，但随着对话越来越长、代码改来改去、评审意见一轮接一轮，AI 的注意力自然而然地往最近的内容上聚焦。那些最早塞进去的 Rules 慢慢就被"挤"出了核心注意范围。这不是 AI 的 bug，Transformer 架构在长上下文下就是这个特性。



破局：能用机器查的就别靠 AI 记



想明白这一层以后，思路就清晰了。很多 Rules 本质上是可以用机器自动检测的。"XAML 不能有中文"——一行正则就搞定了，根本不需要 AI 去"记住"。

   3.5 实战：怎么启动一次完整的多 Agent 研发



说了这么多架构和迭代，可能会好奇：具体操作的时候到底怎么启动？ 其实不复杂，三步就行。



第一步：准备需求文档



不管是一份正式的设计稿，还是自己整理的 PRD，关键是把"要做什么、约束是什么、大致怎么做"写清楚。不需要很长，但得包含背景目标、功能点、约束条件。比如我们做 Galileo 共享库抽取那次，设计稿大概 287 行，包含了背景目标、接入原则、DLL 设计、V3 和 OutsourceLauncher 各自的接入方式、实施步骤。

   4.4 好的事后验证长什么样



做了几个月下来，我觉得好的事后验证系统得满足这么几条：



覆盖率要高，但别贪全。 不是所有规则都适合自动化。比如"Dispatcher 的正确使用方式"需要理解运行时线程模型，正则扫描根本不靠谱。我们的策略是：能自动化的全自动化（目前 18 条规则中 12 条有对应检查项），不能自动化的靠声明式规则做事前预防。



误报率要尽量低。 误报多了大家就不信任验证结果了。为了压低误报我们做了不少工作：

XAML 中文检测：跳过 XML 注释行（<!--）

MessageBox.Show：排除 MessageDialog.xaml.cs（那是合法的 fallback）

硬编码文字：排除已经用了 Loc.Get() 的行

csproj 同步：通过 verification_baseline.json 的排除名单管理历史遗留



修复闭环得硬。 Error 级别必须修完才能过，不存在"先跳过"。Warning 允许任务继续但必须告诉我有哪些 Warning。修复有约束——不许引入新违规、不许删测试、不许改验证脚本来绕过检查。



基线只升不降。 verification_baseline.json 记着当前测试数量基线（369 个）。这个数字只能往上走。如果 AI 试图通过删测试来"通过"验证，B.3 会立刻拦住。



要有历史记录。 每次跑完自动追加到 verification_history.log，方便回看趋势。

   5.2 一个真实的反馈循环



有一回 AI 在 XAML 里写了 Content="设置"。编译通过了，测试也过了，但某些用户的系统字体配置下会显示乱码。



以前怎么办：在 Rule 里加一条"XAML 禁止使用中文字符"。然后看运气——短任务里 AI 通常能遵守，但任务一长、上下文一深，这条 Rule 就可能被"淡忘"，中文又冒出来了。



现在怎么办：

Rule 保留，但精简到只讲"为什么不能用中文"的核心要点

在 verify_all.ps1 里实现 A.1 检查项：用 rg 正则扫描所有 XAML 文件

严重性设为 Error：检测到就不让过



区别在哪？以前 Rule 是"告诉 AI 不要犯错"，本质是预防，但预防会失效。现在 Script 是"犯了错我一定能抓到"，本质是兜底。预防 + 兜底，两层防线才靠得住。



从此以后，不管是哪个 Agent、哪轮对话、上下文有多长，这条规则都是机械化执行的。我的判断只需要表达一次。



   5.3 人的核心工作：从写代码到写规则



仔细想想，Harness 时代人干的事情，本质上是一种元工程——不是在工程本身上做加法，而是在工程的"运转环境"上做加法。



这和传统软件工程里 DevOps 的演进有点像。十年前大家觉得"写代码才是正事，搭 CI/CD 是额外工作"。后来所有人都意识到，CI/CD 才是让团队能稳定交付的基础设施。Harness 对于 AI 开发，就是 CI/CD 对于团队开发——看着像是"额外的活"，实际上是"让所有其他活都靠谱的活"。



具体来说，我现在的日常工作循环是这样的：每次 AI 翻车，我会先判断这个问题属于哪个层面，然后用不同的手段去堵。经过几个月的沉淀，我发现问题基本可以归为五类，每类对应不同的 Harness 改进动作：



第一类：编码规范违反（Rule + Script）



这是最常见的。AI 用了 MessageBox.Show、写了 C# 8.0 语法、XAML 里出现中文——都属于"有明确对错、可以用正则匹配"的问题。处理方式前面 3.4 节已经讲过了：Rule 保留核心要点做事前提醒，Script 做事后兜底检测。这一类问题解决起来最干脆，一旦加了检查项就再也不会漏网。



第二类：操作流程遗漏（Skill）



有些问题不是"写错了"，而是"忘了做某个步骤"。比如 AI 改完代码忘了编译、编译完忘了跑测试、或者不知道 MSBuild 装在哪条路径上。这类问题靠 Rule 写"记得编译"没什么用——AI 不是不知道要编译，而是不知道怎么编译。解决方案是封装成 Skill：一个 SKILL.md 文件描述"做什么、怎么做、看哪个输出文件判断结果"，配一个 .bat 脚本做实际执行。AI 读完 SKILL.md 就知道该跑什么命令、去哪看结果。我们现在有三个核心 Skill——msbuild-compile（编译）、mstest-run（测试）、post-verify（事后验证），覆盖了 AI 最容易漏掉的三个操作步骤。



第三类：流程交互混乱（SubAgent 定义）



前面 3.3 节讲的"Gate Reviewer 自己改需求自己审"就是典型。这类问题的根因不是某个 Agent 能力不够，而是 Agent 之间的交互规则没定义清楚。修复手段是改 SubAgent 的定义文档——加回退路由、细化操作流程、明确禁止事项。比如我在 Code Reviewer 的定义里加了"需求实现 Review"维度以后，"主流程做完了但边界场景漏了"这个问题就几乎消失了。SubAgent 定义改一次，所有后续任务都受益，这是杠杆率最高的一类改进。



第四类：外部能力缺失（MCP 集成）



有些事情 AI 做不到，不是因为规则没写好，而是它根本接触不到所需要的信息或工具。比如我们项目的知识文档放在公司内网的 iWiki 上，AI 看不到就没法参考。再比如有些验证需要查询外部系统的状态。这类问题的解决方式是接入 MCP（Model Context Protocol）——通过 MCP Server 把外部能力暴露给 AI。我们接了 iWiki 的 MCP，AI 现在可以直接调 getDocument 读取 wiki 页面内容，做需求分析时能引用内部文档，而不是凭空猜测。MCP 本质上是在扩展 AI 的"感知范围"——Rule 管"该怎么做"，Skill 管"怎么操作"，MCP 管"能看到什么、能碰到什么"。



第五类：系统性的流程缺失（新阶段 / 新 Agent）



最少见但影响最大的一类。比如最初我们没有事后验证这个环节——不是忘了，而是当时觉得"编译通过 + 测试通过就够了"。后来发现不够，于是加了整个 verify_all.ps1 和 post-verify Skill。再比如最初没有项目级任务看板，导致每次新需求都从零分析。这类问题需要的不是修补现有的某个 Rule 或 Skill，而是往 Harness 里加一个全新的组件。

越往下走，改动频率越低但影响越深。日常大部分时间花在前两层（加个检查项、调个 Skill），偶尔在第三层做一次大改（重新定义某个 Agent 的操作流程），更偶尔在第四、五层做一次系统性的扩展。



这个循环跑得越多，系统越健壮。而且有个很有意思的正反馈效应：护栏越多，AI 能自主完成的任务范围就越大，我需要介入的频率就越低。半年前我得盯着 AI 改每一个文件，现在一个中等需求丢进去，跑完七个阶段拿到的产出，我只需要在闸门评估和代码评审两个关键点重点扫一眼就行。



说得再远一点，我觉得 Harness 这件事本身，其实是一种高度压缩的知识传递。一个资深工程师十年的经验，不可能用一次对话传给 AI。但如果这些经验被拆成一条条 Rule、一个个 Skill、一项项 Script 检查——"XAML 不能有中文"、"编译要用这个路径的 MSBuild"、"SVN 命令必须带认证参数"——AI 就能在秒级时间里"吸收"十年的踩坑经验。Harness 构建得越完善，AI 的行为就越接近一个遵守团队规范的老手，而不是一个什么都不懂的新人。



这大概就是 OpenAI 那句话想表达的意思："Human taste is captured once, then enforced continuously on every line of code." 我的经验和判断只需要表达一次，之后每行代码都在被守护。

收个尾


回头看从 Prompt 到 Context 再到 Harness 这一路走来，最大的感触其实挺朴素的：



AI 编程的瓶颈从来不在 AI 有多聪明，而在我有没有给它搭好发挥的舞台。



好马需要好鞍。规则是缰绳，验证是刹车，Agent 体系是分工——这些东西到位了，AI 才能真正跑起来。缺了任何一环，能力再强的模型也容易跑偏。



这几个月下来我自己最明显的变化是心态：以前 AI 犯错我会觉得"它不够聪明"，现在我会先想"是不是我的 Harness 漏了什么"。一旦养成这个习惯，每一次 AI 的失误都变成了改进系统的机会，而不是重复的挫败感。Harness 这套东西，本质上就是在持续迭代中长出来的——踩一个坑，补一道墙，日积月累，系统就越来越靠谱了。



好马配好鞍，才能走得远。
