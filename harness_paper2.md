一、Harness是什么





Harness这个英文单词按照剑桥词典给出的定义，应当解释为“一种带有带子和腰带的设备，用于控制或固定人、动物或物体”，如果直接用大模型翻译，它可能会告诉你这叫马具、安全带或脚手架。其实这些翻译想表达的意思都是一样的，就是辅助某个“东西”的器具，而这个“东西”就是大模型。



在前一篇公众号文章中，介绍了智能体的基本概念，其中谈到，智能体就是大模型驱动的一种能够自主进行规划、执行、工具使用，并具备一定记忆能力的软件系统。如果把大模型从智能体中剥离出来，就可以得到一个等式：智能体 = 大模型 + harness [1]。换句话说，之前谈到的一切工具、记忆等等智能体中不属于大模型的那部分东西，都属于harness（是的，OpenClaw也属于harness工程的一种实践）。既然并不是什么全新的概念，有读者自然要问了，为什么要造这样一个新词出来呢？



其实，结合OpenAI和Anthropic的说法以及我自己的一些理解，harness的概念要比之前提到的工具和记忆更广，它是为了让智能体能够完成任务（尤其是长周期任务）所必须准备的一整套兜底机制。有些对智能体不是很熟悉的读者可能不太理解，智能体为什么需要额外的机制来保障任务完成呢？



以之前介绍的OpenClaw为例，如果让OpenClaw协助调研一个主题，那么在这一过程中，它可能需要与用户对话以明确调研方向、可能需要上网去收集资料、可能需要摘要工具对检索到的文档进行内容提取、可能需要上下文压缩机制保证整个流程不会超出模型窗口长度。那么从用户发出指令到最终生成回复的过程中，大模型需要用到的所有工具、机制（比如上下文压缩）都属于harness的范畴，如果这些辅助的harness没有准备好，那么这一任务将会因为某种原因无法完成。因此，我把harness定义为辅助智能体完成任务的一整套兜底机制。

二、Claude Code是如何构建Harness的





不知道读者中有多少人是程序员，对Claude Code了解有多少。Claude Code是Anthropic公司基于Claude模型开发的一套辅助编程工具。使用Claude Code，用户可以以自然语言输入的形式让大模型参与代码编写的过程。比如，用户可以输入“请分析本项目所有代码，为我撰写一个Readme文件”，Claude Code就会阅读用户的整个项目，并按照要求写出一个Readme文件。再比如用户可以输入“请把XXX功能加入到当前项目中”，Claude Code就会分析当前的项目状态，然后提出方案，与用户确认后进行代码编写。可以说，熟练使用Claude Code可以显著加速程序员的开发速度。



当然，Claude Code也不是万能的，经常使用的朋友可能会发现，它写的代码也存在问题，甚至是一些人类编写过程中不会出现的问题。身边不少程序员的共同感受是，使用了Claude Code后，大量的时间从写代码变成了审阅代码。实际上，看懂、理解并察觉Claude Code生成的代码中的错误和问题，并不是一件简单的事，因为这份代码不是由自己从头编写的。也正因为如此，我非常鼓励身边的朋友们多用AI工具来辅助自己工作，因为只有多用这些工具，才能知道这些工具擅长做什么、适合做什么，把需要人类的事情留给人类来做，机器能做的事情通通丢给机器去做，工作效率自然就会提升很多。



下面将从七个方面，以Claude Code的设计为基础，来看看真实产品级别的harness是如何构建的。需要注意的是，我没有对源码进行非常细致地阅读，部分资料来源于互联网[2]以及Claude阅读源码后的解释（是的，它是可以看懂自己的源码的）。鉴于本文的重点是让读者了解harness是如何工作的，因此对Claude Code的原本设计也进行了简化，以便于介绍。强烈建议大家有时间去看一看源码，其中有非常多非常精细化的设计，对我们更好地理解智能体非常有帮助。





系统主循环





Claude Code作为一个编程智能体，其本质上还是一个智能体，因此系统主循环与前一篇文章中介绍的ReAct工作模式没有任何区别，都是遵从“用户输入 → 上下文组装 → 模型决策 → 工具执行 → 结果注入 → 继续/停止”这样一个循环的过程。



独特的是，Claude Code将系统主循环拆成了两个层级：查询引擎和查询本身。查询引擎的作用是管理用户的会话，并将合适的上下文组装好交由查询模块进行循环处理。可以简单理解为，在传统ReAct循环的上面又套了一层，专门管理用户的多轮对话信息。每一轮用户的输入都由查询模块循环执行完成，并将相关信息返回给查询引擎进行管理。



查询循环的执行过程也额外设计为了异步生成器的方式，这种设计的好处是不用等到所有内容全部生成结束才返回结果，而是边执行边将可以返回的部分以yield的方式返回，调用方就可以获得实时输出。此外，Claude Code还将工具调用也设计为了流式输出，也就是说，模型只要在生成过程中出现了完整的工具调用，该工具就会被立刻解析并分发执行，等到生成内容完全出现后，工具调用的结果可能已经出现了，大大优化了用户的体验。



在Claude Code的执行循环中，也设计了许多保障循环能够平稳进行的机制。例如，在处理用户的查询前，Claude Code会先预估当前轮的上下文是否可能超出了阈值，如果当前上下文过长，会进入压缩机制以控制整体上下文的长度（下一节中详细介绍）。之后，Claude Code才开始构建真正输入给大模型的上下文。



之前的文章中介绍过，在传统的ReAct循环中，模型继续循环的条件是模型的输出中包含工具调用，此时表明模型尚未完成任务，需要继续。而在Claude Code中，除了工具调用外，还设计了其他继续循环的条件。例如，如果上下文超长了，调用大模型返回报错，则会触发提前设计好的压缩机制对上下文进行进一步的压缩，然后再重新发送给大模型进行处理。



从上面的描述中，想必大家已经能够感受到，即使是对于最简单的ReAct循环，Claude Code仍然设计了非常多的harness，以保障任务的成功执行。而这其中，上下文超长触发压缩后重新发送的设计，就是我所说的一种兜底机制，这种机制会让用户误以为模型都顺利完成了所有任务，而实际上是因为失败被harness“兜”住了，并重新调用了一次大模型，才最终实现了“成功”的效果。这种通过精细化设计保障用户侧任务成功率的设计，也正是harness的精髓所在。





上下文管理





上下文管理是Claude Code中非常重要的一个部分，因为模型的执行效果完全依赖于上下文的内容。众所周知，每个大模型都有一个固定大小的处理窗口，例如Claude最新的模型就支持了100万tokens的输入（注意：这并不意味着100万以内的长度模型都能完美处理，之前的文章中说过，模型的处理能力会随着窗口的增长出现下降）。100万tokens看起来很多（想象一下一本几十万字的小说有多厚），但在代码智能体中，几十次文件读取、数百次工具调用，产生的原始文本很快就会超过100万。因此，哪些东西该保留在上下文中，哪些东西该被压缩，哪些东西该被直接抛弃，是harness必须要解决的问题。



除了效果上的考量，上下文管理还涉及到效率的问题。回顾一下，在使用智能体时（如之前介绍的OpenClaw），每次发送给模型的输入中其实都包含了大量的系统提示词、工具定义等内容，如果每次模型都要从头开始处理这些内容，那无疑会带来巨大的开销（时间&计算）。如何应对这一问题呢？使用过大模型API服务的朋友会发现，一般厂商的计费标准里都会分开列出输入命中缓存的价格和没有命中缓存的价格，且后者会比前者高很多。所谓的命中缓存，其实就是指，对于提示词中完全相同的前缀，模型不用每次都计算一遍（因为计算结果是完全一样的），可以直接复用之前计算的结果。因此，对大模型的上下文管理，还必须考虑到怎样最大化利用前缀缓存。这种缓存的使用是非常严格的，即便前缀中出现了一点点变化，缓存都无法再复用了（因为计算结果不一样）。Claude Code的上下文管理中考虑了许多效果和效率的平衡。



在缓存方面，Claude Code对一部分系统提示词进行了缓存，包括身份信息（“你是Claude Code…”）、系统规则、任务执行指南、操作指南、工具使用规则、语气风格、输出效率指南等。这部分内容对同一用户的每次请求来说都是一样的，因此会在全局进行缓存。系统提示词中还存在另一部分不被缓存的动态内容，包括记忆、用户语言偏好、自定义输出风格等，这部分每个项目和每个用户都不相同。



除缓存以外，上下文管理最为重要的任务是决定哪些内容应当留在上下文中。在这一部分，Claude Code设计了一个五级的压缩机制，一层一层根据需要对上下文进行压缩：

第一层是对工具结果的压缩。这一层实际包含两个粒度的控制：单个工具结果级别和单条消息级别。在单个工具结果级别，如果某个工具返回的结果超过其阈值（默认为5万字符），则直接将该结果以文件的形式存储到本地磁盘，仅在上下文中保留前2000字节的内容预览以及原始结果的存储路径。在单条消息级别，即使每个工具结果都未超过单个阈值，但当同一轮中多个并行工具的结果总和超过消息级预算（默认20万字符）时，系统会从最大的结果开始逐个持久化到磁盘，直到总量降到预算以内。这种压缩方式完全不依赖大模型调用，而是直接对字符串进行操作。在实现压缩的同时，由于原始内容完整保存在磁盘上，模型后续可以通过文件读取工具按需恢复完整内容，兼顾了压缩效率与信息可恢复性。

第二层是对历史消息的裁剪。当上下文增长到一定程度时，系统会从对话历史中裁剪掉较早的、与当前任务关联性较低的消息片段。与工具结果压缩类似，裁剪操作同样不依赖大模型调用，而是直接删除消息。

第三层是微压缩。微压缩专门针对特定类型工具（如文件读取、Shell命令、搜索、网页抓取等）的历史结果进行清理，将较旧的工具返回内容替换为一个简短标记（[Old tool result content cleared]）。

第四层是上下文折叠。上下文折叠是一种比微压缩更为激进但比全量摘要更为精细的压缩方式。它将一段连续的对话轮次（包括之前模型返回的消息、工具调用和工具结果）整体替换为一条由模型生成的摘要消息。

最后一层是全量压缩。这是最后一道主动防线。当前述所有层级的压缩仍然无法将上下文控制在阈值以内时，全量压缩会启动一次完整的模型调用，将整个对话历史压缩为一段摘要。具体实现是通过创建一个子智能体，让其阅读完整的对话历史并生成结构化的摘要，然后用摘要消息替换所有历史消息。压缩完成后，系统还会重新注入关键的上下文附件，例如模型之前读取过的重要文件内容、活跃的子智能体状态、研究计划等，以确保摘要后的模型不会丢失关键的工作上下文。此外，全量压缩还设有熔断机制：如果连续失败超过3次，将停止重试，避免在上下文无法恢复的情况下反复浪费API调用。

可以看到，Claude Code对上下文管理的精细程度达到了非常高的水准。整个压缩体系遵循的核心设计原则是渐进式降级：优先使用不依赖大模型的低开销操作（字符串截断、内容清理），在不得已时才调用大模型进行摘要压缩。每一层都尽可能保留更多的细粒度上下文，只在前一层无法满足需求时才让更激进的后一层接管。



不知道读者有没有注意到一个问题，上述五层压缩流水线中，多个层级（例如全量压缩）的触发都依赖于一个核心判断：当前上下文到底用了多少token？有读者可能要问，多少token还计算不清楚吗？是的，真的计算不清楚，因为对于token的计算是在本地完成的，而绝大部分大模型厂商都不会向用户提供分词器（tokenizer）和词表。因此，当前上下文有多少token，只能通过估算完成，真正用了多少token只有服务端才清楚。Claude Code对当前上下文的估算基于之前一轮模型返回的生成结果，前一轮的API调用中会包含精确的token统计，因此，Claude Code要做的是在前一轮的基础上估算出当前轮新增了多少内容。这部分估算也非常简单，就是用四个字符约等于一个token实现的。可以看到，这套估算的方式本质上也只是一种近似，字符数与token数之间的比率显然是与语言、内容类型、特殊符号等因素高度相关的。而到底上下文有没有超长，只有把消息发送给大模型，看到返回后才知道。因此，这也是为什么在前一节系统主循环中，我们提到上下文超长的错误会触发模型重新进入ReAct的循环。





工具系统





（1）工具的定义与注册

Claude Code 的工具系统是其与外部环境交互的核心接口。之前OpenClaw的文章中已经介绍了，模型本身不能直接操作文件系统、执行命令或访问网络，所有这些能力都通过工具调用来实现。工具系统的设计需要同时满足多个约束：工具集合必须是动态可扩展的（用户可接入自定义工具）、工具定义必须对提示词缓存友好、工具执行必须支持安全管控（用户可对每个工具设定允许、拒绝或询问策略）。



每个工具本质上是一个符合工具接口的对象，包含名称、输入参数的格式、描述文本、权限检查逻辑、以及实际的执行函数。系统提供了一个函数来简化定义过程，它为常见的方法提供安全的默认值（如默认工具不支持并发、默认工具会产生副作用等），开发者只需覆盖与默认行为不同的部分。



工具注册采用集中式清单模式。一个核心的函数枚举了系统所有可能的内置工具，其中许多工具是条件性加载的，根据环境变量等决定是否包含。最终生效的工具列表经过另外一个函数的过滤，再与用户提供的外部工具进行合并，形成完整的工具池。



（2）默认工具集

Claude Code 的内置工具可以大致分为以下几类：

文件操作：Read（读取文件）、Edit（基于差异的精确编辑）、Write（整文件写入）、NotebookEdit（Jupyter 笔记本操作）、Glob（文件名模式匹配）、Grep（内容搜索）

命令执行：Bash（Shell 命令执行，是功能最强大也最需要安全管控的工具）

网络访问：WebSearch（网页搜索）、WebFetch（获取并解析网页内容）

任务管理：Agent（启动子代理执行复杂子任务）、TodoWrite（任务列表管理）

交互与规划：AskUserQuestion（向用户提问）、EnterPlanMode/ExitPlanMode（进入和退出规划模式）

扩展机制：Skill（调用预定义技能脚本）、ListMcpResources/ReadMcpResource（访问 MCP 服务器资源）、ToolSearch（动态发现延迟加载的工具）

除内置工具外，用户通过 MCP 协议接入的外部工具会在运行时动态注入到工具池中，与内置工具享有同等的调用机制和权限管控。



（3）工具调用流程

当模型在生成的响应中包含一个工具调用请求时，系统会对其进行一系列处理后才实际执行：

工具查找：根据工具名称在当前工具池中查找对应的工具定义。若找不到，还会检查是否为已更名工具的旧别名（向后兼容）。

输入校验：验证模型生成的参数是否符合预期类型和结构。模型在生成工具参数时并不总是严格遵循规则（例如将数组写成字符串），这一步会捕获此类错误并将格式化的错误信息反馈给模型。

工具特定验证：调用工具自身的验证输入方法进行逻辑层面的校验（如文件路径是否合法、命令是否在允许范围内）。

权限检查：根据工具类型、用户配置的权限规则、以及自动模式下的安全分类器综合判断是否允许执行。对于需要用户确认的操作，会弹出交互式确认对话框。

执行与结果处理：调用工具执行实际操作。执行结果经过格式化后作为注入到对话历史中，供模型在下一轮推理中参考。

工具执行后处理：运行后执行相关的函数，可用于日志记录、结果修改等。



（4）并行执行

模型可以在一次响应中发起多个工具调用。系统通过工具的“是否并行安全”属性来判断哪些调用可以并行执行。调度逻辑将同一轮次中的多个工具调用分割为若干批次：连续的并发安全工具（如多个文件读取、多个搜索）被归入同一批次并行执行；非并发安全的工具（如文件写入、命令执行）则单独成批串行执行。



此外，系统还提供了一个流式工具执行器，允许在模型响应仍在流式传输的过程中就开始执行已解析完成的工具调用，即不必等待整个响应完成。这种流水线式的并行进一步减少了端到端延迟。



（5）工具搜索与延迟加载

当用户接入大量工具时，将所有工具的完整内容都放入系统提示词会显著占用上下文窗口。为此，系统引入了工具搜索（Tool Search）机制，当模型需要使用某个延迟加载的工具时，它先调用“工具搜索”工具通过关键词搜索或直接名称选择来“发现”目标工具，系统随后将完整的工具定义注入到后续的 API 请求中。



（6）工具结果的渲染与折叠

在用户界面层面，工具调用和结果的展示也经过精心设计。每个工具定义了自己的渲染方法，其可以控制“正在执行什么”的展示、结果的展示以及控制执行过程中的进度反馈。



对于连续的搜索和读取类操作（如多次 Grep、Glob、Read），系统会将它们折叠为一条摘要信息，避免大量重复的工具调用信息淹没用户视野。折叠的判断依据是工具自身的属性，搜索工具（Grep、Glob、Bash 中的 grep/find 命令等）和读取工具（Read、Bash 中的 cat/head 命令等）都会被纳入折叠范围。同类工具的多个并行调用（如同时读取多个文件）还会被预先分组，进一步简化展示。用户可以通过切换详细模式查看被折叠的完整内容。



通过上述内容可以看到，尽管工具系统已经在智能体领域非常常见，其中仍然存在着大量需要精细化设计的部分。而正是这些极致的细节设计，既保证了工具最终执行的成功率，也在用户侧提供了良好的交互。





Hooks设计





Hook 是 Claude Code 提供的事件驱动扩展机制，可以将自定义逻辑注入到智能体运行的关键节点。其设计理念类似于 Git Hooks，在特定事件发生时自动触发预先配置的外部程序，但覆盖范围更广、交互协议更丰富。Claude Code自身的许多功能就是通过hook机制来实现的。



工具生命周期事件是使用最频繁的一类，通过工具执行前触发的hooks，可以检查工具输入并做出允许、拒绝或修改输入的决策；而通过工具执行后触发的hooks，则可用于审计、结果检查或注入额外上下文。



在Claude Code中最重要的一个hook是Stop，它会在模型生成的响应中不包含工具调用（即模型认为任务已经完成）时触发。Stop hook 可以检查最终回答的质量并返回阻塞性错误，迫使模型继续工作。这是实现“质量控制”的关键机制。这也是为什么用户总觉得Claude Code的执行成功率很高，因为不满足质量控制需求的回复会被hook机制拦截并强制模型重新思考，完善回复。



除此以外，Claude Code的记忆机制也是基于hooks实现的。在 Stop hook 阶段，系统会启动记忆提取（从对话中提取值得持久化的信息）和自动复盘（auto-dream）流程，这些任务不会阻塞用户交互。





多智能体系统





Claude Code 的多智能体架构并非单一模式，而是根据任务复杂度和协作需求提供了三种层次分明的协作范式：子智能体派生、协调器模式和智能体团队。三者在自主性、通信方式和上下文管理上各有侧重，形成了从轻量级到重量级的完整谱系。

（1）子智能体派生

第一种子智能体派生形式最为简单，遵循“派生-执行-回收”的单向生命周期。父智能体通过智能体工具创建子智能体，子智能体独立完成任务后将结果返回给父智能体，整个过程不涉及双向通信。



系统预定义了多种子智能体类型，每种类型针对特定场景优化。例如通用型智能体拥有完整工具集，适合研究和实现等复杂任务；探索型智能体专注于代码库快速搜索，仅保留读取类工具；规划型智能体用于设计实现方案，禁用所有写入操作。每种类型通过工具白名单、系统提示词和最大轮次等参数进行差异化配置。



子智能体的创建过程包含五个阶段：首先根据类型名称解析具体的智能体定义；然后组装该类型允许使用的工具池；接着构建包含角色指令的系统提示词；随后创建隔离的执行上下文；最后启动实际的推理循环。其中上下文隔离是关键的安全设计，子智能体默认无法访问父智能体的任何状态，需要通过显式的配置来共享特定资源。



一个特殊的变体是Fork模式。与常规子智能体从空白上下文开始不同，Fork子智能体完整继承父智能体的对话历史和系统提示词。这一设计的核心动机是提示缓存共享：所有从同一时刻派生的Fork子智能体拥有字节级相同的API请求前缀，从而在大语言模型侧命中缓存，显著降低推理成本。为实现这一点，系统在构建Fork对话时，会将父智能体的所有工具调用结果替换为统一的占位符文本，仅在末尾附加各自不同的任务指令，确保除最后一个文本块外，所有内容完全一致。Fork子智能体还内置了递归保护机制，通过检测对话历史中的特定标记来阻止Fork子智能体再次派生新的Fork。



子智能体的权限处理也值得关注。系统提供了多种权限模式：子智能体可以将权限请求冒泡到父终端让用户决定，也可以在受限环境中自动拒绝所有需要权限的操作。这使得子智能体既能执行需要用户授权的敏感操作，也能在完全自动化的流水线中安全运行。

（2）协调器模式

协调器模式将父智能体的角色从“执行者”转变为“指挥者”。在这种模式下，父智能体（协调器）本身不执行任何具体的编码或文件操作，而是专注于理解用户意图、分解任务、分配工作、综合结果。



协调器拥有精简的工具集：仅保留派生工作者、发送消息、停止任务等管理类工具。所有实际的代码读写、命令执行等操作都委托给工作者智能体完成。工作者智能体则拥有完整的标准工具集和 MCP 工具访问权限，能够独立完成研究、实现和验证等任务。



这一模式定义了清晰的四阶段工作流程：研究阶段由多个工作者并行调查代码库和问题背景；综合阶段由协调器阅读研究结果，理解问题本质，制定具体的实现方案；实现阶段由工作者按照协调器给出的详细指令执行变更；验证阶段由独立的工作者测试和验证变更效果。协调器在综合阶段承担最关键的职责——它必须真正理解工作者的研究发现，而非简单转发。系统提示词中明确禁止“根据你的发现修复这个问题”这类委托式指令，要求协调器在指令中包含具体的文件路径、行号和修改内容。



协调器与工作者之间的通信采用异步模式。工作者的结果以任务通知的形式异步送达，协调器在等待期间可以继续响应用户或启动新的工作者。这一机制使得并行处理成为协调器的核心优势——独立的研究任务可以同时展开，多角度覆盖问题空间。协调器也需要管理并发冲突：只读的研究任务可以自由并行，但涉及文件写入的实现任务需要避免操作同一组文件。



协调器还支持对工作者进行生命周期管理，包括在方向错误时及时停止工作者、通过追加消息继续已完成工作者的上下文、以及根据上下文重叠度决定复用工作者还是创建新工作者。

（3）智能体团队

智能体团队是最复杂的多智能体模式，实现了真正的对等协作。与前两种模式中子智能体由父智能体单向管理不同，团队中的每个成员都拥有独立的会话和收件箱，能够相互发送消息，形成去中心化的协作网络。



团队的创建从领导者开始。领导者通过专门的团队创建工具初始化团队，生成团队配置文件并注册自身为首个成员。随后，领导者可以派生新的队友加入团队。每个队友拥有独特的身份标识（由名称和团队名组合而成）、分配的颜色（用于 UI 区分）以及独立的工作目录。



团队成员之间通过信箱系统进行异步通信。每个成员拥有一个独立的收件箱，其他成员通过消息发送工具将消息写入目标成员的收件箱。收件箱基于文件系统实现，支持点对点消息和全队广播两种模式。除普通文本消息外，信箱还承载结构化的协议消息，例如关闭请求与响应、方案审批与驳回等，实现了完整的生命周期协调。



团队模式还引入了共享暂存区的概念，这是一个所有成员都可以自由读写而无需权限确认的目录，用于跨成员的知识持久化。成员可以将研究发现、中间结果或协调信息写入暂存区，其他成员随时读取。这弥补了异步消息在传递大量结构化数据时的不足，为团队协作提供了共享的持久化存储。



以上就是Claude Code中的多智能体系统。三种模式形成了递进的复杂度阶梯：子智能体适合简单的任务分解和并行执行；协调器适合需要统一规划和分阶段推进的复杂任务；智能体团队则适合需要成员间频繁交互和长期协作的大型项目。这些模式的选择完全是由模型根据系统提示词中的引导自主判断的。





记忆系统





Claude Code 的记忆系统解决了智能体的核心困境：没有记忆，每次会话都从零开始，用户的偏好、项目背景和历史纠正全部丢失。该系统的核心想法是只记忆无法从当前项目状态推导的信息。也就是说，代码模式、架构、文件结构和 Git 历史这些自描述的信息，直接读取永远比从记忆中回忆更准确。



用过Claude Code的朋友可能对CLAUDE.md比较熟悉，该文件是由用户手动维护的静态配置文件，通常包含项目规范和编码规范。这与记忆系统的功能是不同的。记忆是由系统自动写入或通过命令触发的动态知识库，存储个人偏好、行为纠正和项目动态。CLAUDE.md会在每次会话中全量加载；而记忆系统则采用索引预加载和按需语义召回的机制，在保证信息可达性的同时控制上下文占用。



Claude Code中的记忆按照类型可以分为四种：

用户记忆：记录用户的身份、角色、偏好和知识背景，帮助智能体在未来的交互中实现个性化服务。例如“用户是数据科学家，关注可观测性”。

反馈记忆：记录用户对智能体行为的纠正和肯定。这类记忆会以结构化的方式记录，包含规则本身、“为什么”和“如何应用”三个部分。系统同时记录正面和负面反馈，因为如果只保存纠正，智能体会避免过去的错误，但可能偏离用户已经验证过的好方法。记录“为什么”使得智能体在面对边缘情况时能做出判断，而非机械执行规则。

项目记忆：记录项目进度、决策和截止日期等动态信息。要求将用户消息中的相对日期转换为绝对日期（如将“周四”转为“2026-04-09”），因为记忆可能在数周后才被读取。

参考记忆：记录外部系统的位置指引，如“流水线缺陷跟踪在 Linear 的 INGEST 项目中”。

系统同时维护一个明确的排除清单：代码模式、架构、文件路径、Git 历史、调试步骤、CLAUDE.md中已有的内容以及临时任务细节，即使用户明确要求保存也不会写入记忆。



记忆以独立的 Markdown 文件形式存储在项目目录下，每个记忆文件包含元数据（名称、描述和类型）。其中描述字段至关重要，它是后续语义召回时评估相关性的主要依据，因此必须具体。



目录结构中还有一个 MEMORY.md 索引文件，作为记忆系统的入口点。它既是所有记忆文件的索引，又是每次会话启动时加载到上下文中的快速参考。由于它每次会话都会加载，因此必须保持紧凑。该文件中的每条索引应为一行链接加简短描述。



Claude Code的记忆召回是通过一个Side-query模型实现的（通常是Sonnet模型）。通过将所有记忆文件中的前面部分的内容提供给Sonnet模型，让其选择与当前查询最相关的记忆（最多5条）。



记忆的新鲜度管理是防止过时信息误导模型的关键机制。系统将修改时间转换为人类可读的时间距离（如“今天”、“昨天”、“47 天前”等），因为模型不擅长日期算术。“47 天前”这样的描述能直接触发模型的“这可能已过时”推理。对于超过 1 天的记忆，系统会注入警告：“关于代码行为或文件引用的声明可能已过时”。



除了用户显式保存和主智能体主动写入之外，系统还运行一个后台提取智能体，在主智能体完成响应（无后续工具调用）后自动从对话中提取值得记忆的信息。







规划（Plan）模式





计划模式是 Claude Code 中唯一一个模型主动降低自身权限的机制。其核心设计哲学是：对于复杂任务，模型先探索、再规划、最后在用户批准后才执行。如果没有这一机制，一个“重构整个认证系统”的请求可能导致模型立即修改十余个文件、删除多个函数并引入不必要的依赖，迫使用户回滚所有变更从头来过。



计划模式有两条进入路径。用户可以通过命令主动触发，也可以由模型在评估任务复杂度后自主调用进入工具。模型自主进入的判断依据包括：涉及新功能实现、存在多种可行方案、需要架构决策、变更影响多个文件、需求不够明确等情况。



进入计划模式时，系统执行对称的状态转换：记录进入前的权限模式（称为“前置模式”），然后将当前模式切换为计划模式。所有写入操作被阻止，模型只能使用读取类工具探索代码库。这种设计确保了退出时能够精确恢复原始模式，使计划模式成为一个无副作用的可嵌套插入层。



计划模式的默认工作流程分为五个阶段：初始理解阶段派发探索型智能体调查代码库；设计阶段派发规划型智能体构思解决方案；审查阶段由主智能体综合结果并向用户提出澄清性问题；编写阶段将最终方案写入计划文件；退出阶段调用退出工具请求用户审批。



系统还提供了一种迭代式工作流的替代方案：不是一次性完成所有探索后再编写计划，而是在探索、更新计划文件和询问用户之间形成持续循环，鼓励快速交互而非穷尽式的前期探索。



退出计划模式时，系统首先验证模型确实处于计划模式。这一检查之所以必要，是因为模型有时会“忘记”自己已经退出，比如上下文压缩可能移除退出信号。用户可以直接批准计划、编辑后批准或者拒绝（继续留在计划模式）。被编辑过的计划会被标记为“用户已编辑”，提醒模型注意变更。



三、Harness在深度搜索中的实践





在学习了Claude Code的设计后，我产生了一个非常有趣的想法。因为我本身是做搜索相关研究的，我想知道如果从harness工程的角度（借鉴Claude Code的一些思想）来构建一个深度搜索（或叫深度研究）的智能体，会产生什么样的效果。于是我从头构建了一个深度搜索智能体的harness，叫SearchClaw [3]。

图片
图5：SearchClaw的结构图



如图5所示，我也从研究计划制定、上下文管理、基于hook的质量控制和记忆等角度构建了深度搜索的系统。下面简单介绍一下在实现这一系统的过程中，我的一些想法和设计。



首先是研究计划，这个模块的想法我一直都有，刚好Claude Code中也有一个类似的to-do list功能，这一功能的实现给了我一些启发。我认为深度搜索或深度研究任务中，一个好的计划是非常重要的。计划的制定有利于模型在任务开始前对整个任务有个全局的了解。此外，随着任务的推进，研究计划应当对计划中的每个步骤进行一些总结性的记录，以便于智能体了解当前任务的执行进度。对于深度搜索这样较为长程的任务来说，一个计划板可以很好地帮助模型避免陷入局部搜索之中。在设计这一模块的时候，我还考虑到问题本身潜在的模糊性问题。因此，在制定计划之前，我为系统设计了一个查询澄清模块，当模型（采用的也是side-query model）判定用户的输入较为模糊时，会自动根据可能的调研方向弹出几个选项供用户选择，也允许用户自行输入澄清内容。通过这样的设计，系统可以更加理解用户的真实意图，并提供更加丰富多样的信息。



搜索和阅读是深度研究智能体两个非常重要的模块。在搜索方面，我为系统设计了多级兜底机制。系统将会首先使用搜索API（例如Serp等），如果API因为各种原因无法访问（例如超出额度），则会由浏览器搜索进行兜底（依赖于playwright或CDP），如果浏览器也无法访问，则会使用duckduckgo搜索引擎进行直接请求。这样三级兜底机制基本上可以保证搜索工具能够返回结果。除了兜底以外，我也同样为系统配置了学术搜索、新闻搜索和公众号搜索等其他工具，以便于系统能够获得更加多元化的信息。在阅读方面，我也模仿Claude Code的设计做了一整套的压缩机制。首先，将访问的网站内容缓存到磁盘上，然后，根据实际内容大小决定是将全文送入模型还是使用side-query model对内容进行压缩。如果压缩了，也同样会为模型提供原始文件的访问路径。这样的一套机制可以有效缓解模型的上下文压力，防止上下文被外部检索结果填满。



除了对阅读工具的结果进行长度控制外，我也设计了一套完整的上下文压缩机制。其主要分为微压缩和全压缩两部分。微压缩主要对工具调用的结果进行压缩，如果不能满足压缩要求，则使用side-query model对上下文进行完整的压缩。这套机制可以保证模型在进行多轮工具调用的过程中，始终将上下文控制在合理的范围内。



在质量控制部分，我采用了类似Claude Code的hook机制。在模型返回没有工具的结果时，会对这一结果进行多方面的校验：生成的结果是否包含多个引用、引用的来源是否多样、生成的结果长度是否足够以及之前制定的研究计划中的内容是否全部都完成。一旦有一项校验失败，生成结果会被重新退回给模型，要求其按照要求改进后再回复。在实际使用中，这套机制很好地保证了生成结果的质量。



而在记忆模块，我也充分考虑了深度搜索任务的特性。我会在记忆中重点记录哪些网站来源是较为官方的，以便于之后在进行阅读时优先考虑这些网站的内容。对于曾经访问过的网站也会有一些记忆，以便于对之后进行相似问题调研时提升效率。



在这样一整套机制的保障下，SearchClaw确实可以取得非常好的搜索质量。



尽管这一系统已经能够平稳运行，但我仍然有不少顾虑。首先是我并不知道这些设计中哪些是真正有用的。跑基准测试其实对于这类系统的评估是有偏的，我在系统中设计的一些面向用户的设计（例如询问用户问题），显然没有办法在基准测试中发挥作用，但我认为它们还是能有效提升用户体验的。其次是每个模块的功能很难进行定量分析，模块之间的相互作用也难以探究，比如阅读部分的压缩和上下文管理机制中的压缩应该采用什么样的协同机制，其实是很难通过实验不断调试出来的。



但不论如何，借助于Claude Code中的一些成熟设计，整个系统体验下来的流畅性和成功率都非常让人欣喜。我想这就是harness工程的重要意义。



四、对harness的一些理解和想法





经过前文的介绍，相信大家对harness的概念有了一定的了解。而我本人通过SearchClaw这样一个项目，也对harness的设计有了更加深入的体会。下面还是用自问自答的方式简单谈一谈我的理解和想法。

问

Harness的意义是什么？

答

目前看来，任务成功率与模型和harness都高度相关，完成同样的任务，越强的模型，harness就可以越轻量化，而越弱的模型，则越依赖于harness的设计。不知道大家有没有在官网使用过Claude模型，Claude 官网的体验中已经融入了不少智能体的设计理念。比如让其完成一些任务的时候，它会自动进行工具调用。也就是说，它可能也有一套完整的harness来保障任务的成功率，这也是为什么Claude的使用体验比其他家要好很多的原因。其实很多时候可能Claude也没有生成足够好的答案，但因为harness的兜底机制，导致其产生的一些错误被阻挡住了，从而修正了最终的答案。因此，我认为未来除了要提升基础模型的能力，也要为其实际应用构建最为适合的harness。我相信Claude Code在Claude模型本身更新后又会发生巨大的变化，原来需要harness兜底的问题可能模型能力提升后就不需要了，就像随着工程的进行，“脚手架”被逐渐拆除一样。



问

Harness未来的可能发展是什么？

答

从最近的一些工作（比如Meta-Harness[4]）来看，未来一个很重要的方向是如何让智能体能够自主优化自己的harness。从Claude Code和Codex的开发博客中可以看到，两款产品的开发人员都是通过大量使用Claude和GPT的方式来进行开发的。也即是说，现在的这些harness本来也是由模型生成得到的。只不过，现在的这些设计仍然高度依赖于人类的经验。最近还有一些工作在讨论自动skill/tool生成，我认为其本质上也是auto-harness的一种。另外，从OpenClaw的体验中我就直观地感受到，现在的harness过于绑定基础模型了。很多设计都是在单个模型（如Claude）的条件下产生的，这导致对其他模型的适配上必然存在不足。一个针对Claude优化后的harness，极有可能不适用于GPT模型。因此，未来是否存在能够在某些基础上针对接入模型的不同而产生自主适配的harness方法，也是非常有趣的方向。总之，在基础模型真的能够拥有极其强大的能力之前，harness应该是很长一段时间模型所必须依赖的系统工程。它极为重要，但可能最终会消失。



问

现在需要专门学习harness吗？

答

从我个人的感受来说，目前的harness太过于工程化了。很多看起来有道理的设计可能都没有经过细致的实验验证。甚至无法进行实验验证，因为不同设计的组合实在是太多了。我建议对harness感兴趣的同学抽时间把Claude Code的源码仔细阅读一下，绝对受益匪浅。另一方面，从Claude Code的源码设计上也可以看到，现在对上下文管理、记忆管理、工具调用等各种方面的研究广义上来说也属于harness研究的一部分。换句话说，harness 并非一个全新的领域，而是对既有智能体工程实践的一种系统性归纳。
