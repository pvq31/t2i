# seethrough3d2 Harness 化修改方案（2026-05-12）

本文基于 `代码说明.md`、`harness_paper1.md`、`harness_paper2.md`、`agent_text2pkl_v5.py`、`agent_check_pkl_v5.py`、`agent_opinion.py`、`agent_reverse_core.py`、`run_scene_text_pipeline.py` 的现状整理。核心结论是：当前项目已经有多 agent 和 pkl 中间态，但还没有把“文字空间关系”变成独立、可验证、不可绕过的结构化约束。因此 `agent_text2pkl_v5.py` 效果一般的根因不是 prompt 不够强，而是缺少 `text -> predicates -> numeric constraints -> solver/projector -> verifier -> repair log` 这条 harness 主链路。

## 一、8 条 harness 特征逐项检查

### 1. 分层、有明确循环、agent 角色分工

现状：部分做到。

已经存在的分工：
- `agent_text2pkl_v5.py`：文生场景 pkl。
- `agent_opinion.py`：评估 pkl 并输出修改意见和 JSON。
- `agent_reverse.py` / `agent_reverse_core.py`：根据 guidance 修 pkl。
- `agent_check_pkl_v5.py`：评估 + 修复 pkl。
- `pkl2fig_v3.py`：pkl 渲染 cube 图。
- `infer2_v2.py`：pkl + prompt 进入图像生成。
- `run_scene_text_pipeline.py`：串联 text2pkl、check、infer。

主要缺口：
- `run_scene_text_pipeline.py` 是固定顺序，不是由验证结果驱动的状态机。
- `--check-runs` 是人工指定轮数，不是“失败类别 -> 调用对应工具 -> 再验证”的闭环。
- 没有统一 run 目录保存每一轮 plan、predicate、constraint、validation、repair action。

建议实现：
- 新增 `run_scene_harness.py`，作为 PM/Orchestrator 状态机：
  1. `PLAN`：解析文字为对象清单和关系谓词。
  2. `EXECUTE`：生成初始 pkl 或 scene graph。
  3. `CONSTRAIN`：把谓词编译成数值区间，并投影/求解坐标。
  4. `VERIFY`：机器检查对象、关系、物理、尺寸、遮挡、相机可见性。
  5. `REPAIR`：按失败类别调用对应工具修复。
  6. `RENDER`：通过 pkl2fig / infer2 生成可视结果。
  7. `STOP`：全部通过或达到最大轮数。
- 每一轮不再用自然语言“try harder”，而是用 JSON issue 路由：
  - `missing_object` / `extra_object` -> object reconciler。
  - `relation_mismatch` -> constraint projector。
  - `support_mismatch` -> support placer。
  - `collision` -> collision resolver。
  - `frame_clipping` -> camera fitter。
  - `semantic_binding` -> 重新解析对象绑定。

### 2. 有结构化中间状态，而不是黑箱 prompt

现状：部分做到。

已经做到：
- `inference/saved_scenes/*.pkl` 保存 `subjects_data`、`camera_data`、`surrounding_prompt`、`inference_params`。
- `agent_text2pkl_v5.py` 同名导出 JSON，便于查看 pkl 内容。

主要缺口：
- pkl 是结果状态，不是“意图状态”。它没有保存 prompt 中抽出的对象、关系、约束、冲突、修复轨迹。
- 关系仍隐藏在 prompt 或 guidance 中，没有独立的 `left_of(A,B)`、`support(A,B)`、`above(A,B)`。

建议实现结构化中间态：
- 新增 `scene_graph.json`：
```json
{
  "source_text": "...",
  "objects": [
    {"id": "refrigerator_1", "mention": "a refrigerator", "type": "refrigerator", "count_source": "a"}
  ],
  "predicates": [
    {"id": "p1", "type": "centered", "subject": "refrigerator_1", "evidence": "in the center"},
    {"id": "p2", "type": "right_of", "subject": "cabinet_1", "object": "refrigerator_1", "evidence": "to the right of"},
    {"id": "p3", "type": "support", "subject": "microwave_1", "object": "cabinet_1", "evidence": "placed on the countertop"}
  ]
}
```
- 新增 `constraints.json`：
```json
{
  "constraints": [
    {"predicate_id": "p2", "kind": "interval", "var": "cabinet_1.y", "lower": "refrigerator_1.y + ...", "upper": null},
    {"predicate_id": "p3", "kind": "equality", "var": "microwave_1.z", "value": "cabinet_1.z + cabinet_1.height"}
  ]
}
```
- 新增 `validation.json` 和 `repair_log.jsonl` 保存每轮验证和修复。

### 3. 有可自主运行的工具调用，而不是只靠 prompt / guidance

现状：部分做到，但还不够 harness。

已经有工具雏形：
- `agent_check_pkl_v5.py` 中已有 `apply_strict_subject_constraints()`、`build_support_map()`、`enforce_text_relations()`、`fit_scene_to_camera()`、`run_local_checks()`。
- `agent_reverse_core.py` 中有较完整的修复函数。

主要缺口：
- 这些函数没有被包装成清晰的工具 API，agent 不能按能力选择调用。
- `guidance.md` 仍是自然语言操作手册，agent 需要“记住怎么改 pkl”。
- `agent_text2pkl_v5.py` 仍让 LLM 直接输出坐标，再用启发式修正。坐标不是由工具从关系谓词机械生成。

建议新增工具层：
- `harness/tools/object_tools.py`
  - `extract_objects(scene_text) -> object_mentions.json`
  - `reconcile_objects(scene, expected_objects) -> scene`
- `harness/tools/predicate_tools.py`
  - `extract_predicates(scene_text, objects) -> predicates.json`
  - `bind_predicate_args(predicates, objects) -> bound_predicates.json`
- `harness/tools/constraint_tools.py`
  - `compile_constraints(scene, predicates) -> constraints.json`
  - `project_scene_to_constraints(scene, constraints) -> scene`
- `harness/tools/verify_tools.py`
  - `verify_objects(scene, object_mentions) -> issues`
  - `verify_predicates(scene, predicates) -> issues`
  - `verify_physics(scene) -> issues`
  - `verify_visibility(scene, camera) -> issues`
- `harness/tools/repair_tools.py`
  - `repair_by_issues(scene, issues) -> scene, repair_actions`

`guidance.md` 应从“教 agent 如何手改参数”改成“工具语义说明”。机械计算写进 Python，不再依赖模型记忆。

### 4. 工具调用由 agent 觉得需要时调用，而不是流程写死

现状：基本没做到。

当前 `run_scene_text_pipeline.py` 是固定顺序：
`agent_text2pkl_v1.py -> agent_check_pkl.py * N -> infer2_v2.py`。

建议实现工具选择策略：
- 新增 `harness/tool_registry.py`，每个工具登记：
```json
{
  "name": "project_scene_to_constraints",
  "handles": ["relation_mismatch", "support_mismatch", "above_below_mismatch"],
  "input": ["scene", "predicates", "constraints"],
  "output": ["scene", "repair_actions"],
  "safe_to_repeat": true
}
```
- 新增 `harness/policy.py`：
```python
def choose_tools(state):
    if state.object_issues:
        return ["reconcile_objects"]
    if state.predicate_issues:
        return ["project_scene_to_constraints", "verify_predicates"]
    if state.physical_issues:
        return ["repair_physics", "resolve_collisions"]
    if state.visibility_issues:
        return ["fit_camera"]
    if state.image_semantic_issues:
        return ["rerun_infer_with_masks_or_prompt"]
    return []
```

什么时候调用哪些工具：
- prompt 中解析出任何空间关系谓词，就必须调用 `compile_constraints` 和 `project_scene_to_constraints`。
- `verify_predicates` 失败时调用 `project_scene_to_constraints`，不能只让 LLM 重新生成。
- `support(A,B)` 失败时调用 `place_on_support(A,B)`；如果 A 比 B 大，先调用 `shrink_to_support(A,B)` 或返回 `unsat_support_size`。
- 物体数量不匹配时调用 `reconcile_objects`。
- cube 出画时调用 `fit_camera`，不要修改语义关系。
- 关系冲突无法同时满足时返回 `unsat_predicates`，让 parser/agent 重新解释，而不是随意牺牲某条关系。

### 5. 有规则和约束，而且约束能被机器检查

现状：部分做到。

已经做到：
- `agent_opinion.py` 能输出 A-F 标准的结构化 JSON。
- `agent_check_pkl_v5.py::run_local_checks()` 能检查对象数量、未知类型、尺寸、地面、支撑、可见性、部分尺寸排序。
- `agent_opinion.py::evaluate_criterion_b()` 能检查 left/right/front/behind/above/below/support 的一部分关系。

主要缺口：
- prompt 中的规则没有先变成 `predicates`，验证脚本只是从自然语言里再次正则匹配，稳定性有限。
- `agent_text2pkl_v5.py` 的 `repair_subject_layout()` 调用 `apply_strict_subject_constraints(..., guidance_text="")`，而 `enforce_text_relations()` 对 scene_text 主要处理 left/right/front/behind/center，不处理通用 `above/below`；`above/below` 更多在 `apply_manual_relation_guidance()` 中处理，但 v5 生成阶段没有把 scene_text 当 guidance 传入。因此“XX above YY”这类 prompt 很容易不被强制落实。
- 没有把每条规则编译为明确数值区间。

必须实现的机器约束：

坐标约定：
- `x`：前后，越大越靠前。
- `y`：左右，越小越靠左，越大越靠右。
- `z`：上下，越大越高。
- `dims = [width, depth, height]`。
- 当前代码保存 pkl 时：`pkl.x = scene_x - 6.0`，`pkl.y = scene_y`，`pkl.z = scene_z`。约束求解建议统一在 `scene_x/scene_y/scene_z` 上做，最后再写回 pkl。
- 当前几何检查默认 `z` 是物体底面高度，所以物体顶面是 `z + height`。

基础几何：
```python
left_edge(A)   = A.y - A.width / 2
right_edge(A)  = A.y + A.width / 2
back_edge(A)   = A.x - A.depth / 2
front_edge(A)  = A.x + A.depth / 2
bottom(A)      = A.z
top(A)         = A.z + A.height
```

关系谓词必须编译成以下数值条件：
- `left_of(A,B)`：
  - 强条件：`right_edge(A) <= left_edge(B) - margin_lr`
  - 等价区间：`A.y <= B.y - B.width/2 - A.width/2 - margin_lr`
- `right_of(A,B)`：
  - `left_edge(A) >= right_edge(B) + margin_lr`
  - `A.y >= B.y + B.width/2 + A.width/2 + margin_lr`
- `in_front_of(A,B)`：
  - `back_edge(A) >= front_edge(B) + margin_fb`
  - `A.x >= B.x + B.depth/2 + A.depth/2 + margin_fb`
- `behind(A,B)`：
  - `front_edge(A) <= back_edge(B) - margin_fb`
  - `A.x <= B.x - B.depth/2 - A.depth/2 - margin_fb`
- `above(A,B)`：
  - 最低要求：`bottom(A) >= top(B) + margin_z`
  - 也就是 `A.z >= B.z + B.height + margin_z`
  - 如果文字是 “on / on top of / placed on / sits on”，应改为 `support(A,B)`，使用更严格条件。
- `below(A,B)`：
  - `top(A) <= bottom(B) - margin_z`
  - `A.z + A.height <= B.z - margin_z`
- `support(A,B)`：
  - `abs(bottom(A) - top(B)) <= z_tolerance`
  - `left_edge(A) >= left_edge(B) + inset`
  - `right_edge(A) <= right_edge(B) - inset`
  - `back_edge(A) >= back_edge(B) + inset`
  - `front_edge(A) <= front_edge(B) - inset`
- `centered(A)`：
  - `abs(A.x) <= center_tolerance`
  - `abs(A.y) <= center_tolerance`
- `same_support(A,B,S)`：
  - `support(A,S)` and `support(B,S)`，并在同一支撑面上使用较小 lateral/front-back margin。

默认参数：
```python
margin_lr = max(0.04, 0.05 * max(A.width, B.width))
margin_fb = max(0.04, 0.05 * max(A.depth, B.depth))
margin_z = 0.02
z_tolerance = 0.03
inset = 0.02
center_tolerance = 0.15
```

这些规则必须由 `verify_predicates.py` 检查。LLM 只能解释“为什么”和“对象绑定”，不能决定违反规则时算不算通过。

类似更多的取值范围可以见《guidance.md》和《代码说明.md》中的 pkl/json场景文件参数说明,需要全部实现强制机械的pkl参数（例如：inference/saved_scenes/example0.json或inference/saved_scenes/example0.pkl）的取值范围限制，而不是依靠prompt来实现

## 二、`agent_text2pkl_v5.py` 效果一般的原因

### 原因 1：LLM 被要求直接输出坐标，空间关系没有中间谓词层

`build_prompt()` 要求模型直接给出：
```json
{"scene_x": 0.0, "scene_y": 0.0, "scene_z": 0.0}
```

这会让模型同时承担四件事：对象抽取、对象绑定、空间关系解析、数值求解。任何一步偏差都会落到最终 pkl 中，而且后处理不知道模型为什么这么摆。

修改方向：
- LLM 第一阶段只输出对象和关系，不输出最终坐标。
- 坐标由工具根据谓词和尺寸机械求解。

### 原因 2：对象抽取靠 a/an/数字 + 正则，复杂名词短语和并列结构不稳定

当前 `parse_quantified_object_mentions()` 根据 determiner 后面的 token 截断。它能处理很多简单句，但对以下情况不稳：
- “books on the shelf” 没有 a/an。
- “a microwave and a bowl are placed on the countertop” 需要稳定继承同一支撑关系。
- “the countertop of the cabinet” 不是独立资产，但表达了 cabinet 的支撑面。
- “two birds: a pigeon..., a sparrow...” 需要处理类别集合和具体实例。

修改方向：
- LLM parser 输出 `object_mentions`，本地 verifier 再用 asset table 校验。
- 对 `the countertop of cabinet` 编译为 `support(microwave, cabinet)` 和 `support(bowl, cabinet)`，不要把 countertop 当新物体。

### 原因 3：关系修复函数不是完整的谓词验证/求解器

当前 `apply_strict_subject_constraints()` 有实用启发式，但它没有独立的 predicate list。尤其是：
- `enforce_text_relations()` 对 scene_text 主要处理 left/right/front/behind/center。
- `above/below` 的强制逻辑在 `apply_manual_relation_guidance()` 中更完整，但 v5 生成阶段传入的 `guidance_text=""`，因此 prompt 中的 `above/below` 不一定被强制执行。
- 关系检查和关系修复并非同一个 predicate 源头，可能出现“检查能发现，但生成阶段没强制”的断裂。

修改方向：
- 把所有关系先抽成统一 predicate。
- 约束编译、投影、验证都读取同一个 `predicates.json`。

### 原因 4：当前修复是启发式移动，不是参数取值范围求解

例如 `left_of(A,B)` 当前通常根据 gap 调整 `A.y`，但没有显式记录：
```text
A.y <= B.y - B.width/2 - A.width/2 - margin
```

修改方向：
- 每个谓词生成一个或多个参数区间。
- projector 每次改动都记录 old/new/reason。
- verifier 用同一公式复查。

### 原因 5：同类型多实例容易绑定错

当前很多地方用 name/type alias 匹配。如果 prompt 中有 two chairs 或多个同类对象，关系如 “the left chair is behind the table” 需要稳定绑定到实例。

修改方向：
- 对象必须有唯一 `id`，例如 `chair_1`、`chair_2`。
- 谓词参数必须引用 id，不引用自由文本 name。
- 每条 predicate 保存 `evidence` 和 `confidence`，低置信度进入 disambiguation。

### 原因 6：相机可见性和语义关系可能互相干扰

当前 fit camera / shrink objects 可能改善可见性，但如果在关系修复后又整体缩放/移动，可能需要再次验证所有谓词。

修改方向：
- 每次 camera/layout transform 后必须重新跑 `verify_predicates`。
- 若为纯相机问题，优先改 camera，不改 subject 语义参数。

## 三、`agent_text2pkl_v5.py` 具体修改方案

### 目标

把 v5 从“LLM 坐标生成器”改成“LLM 语义解析 + 工具化约束求解器”。

### 新增 `compile_predicate_constraints(scene, predicates)`

这个函数是本次改造最关键部分。它必须机械生成参数区间。

伪代码：
```python
def compile_predicate_constraints(scene, predicates):
    constraints = []
    for p in predicates:
        A = scene[p.subject]
        B = scene[p.object] if p.object else None

        if p.type == "left_of":
            upper = B.y - B.width / 2 - A.width / 2 - margin_lr(A, B)
            constraints.append(interval(p.id, f"{A.id}.y", upper=upper))

        elif p.type == "right_of":
            lower = B.y + B.width / 2 + A.width / 2 + margin_lr(A, B)
            constraints.append(interval(p.id, f"{A.id}.y", lower=lower))

        elif p.type == "in_front_of":
            lower = B.x + B.depth / 2 + A.depth / 2 + margin_fb(A, B)
            constraints.append(interval(p.id, f"{A.id}.x", lower=lower))

        elif p.type == "behind":
            upper = B.x - B.depth / 2 - A.depth / 2 - margin_fb(A, B)
            constraints.append(interval(p.id, f"{A.id}.x", upper=upper))

        elif p.type == "above":
            lower = B.z + B.height + margin_z
            constraints.append(interval(p.id, f"{A.id}.z", lower=lower))

        elif p.type == "below":
            upper = B.z - A.height - margin_z
            constraints.append(interval(p.id, f"{A.id}.z", upper=upper))

        elif p.type == "support":
            target_z = B.z + B.height
            constraints.append(equality(p.id, f"{A.id}.z", target=target_z, tolerance=z_tolerance))
            constraints.extend(footprint_inside_constraints(A, B, p.id))

        elif p.type == "centered":
            constraints.append(interval(p.id, f"{A.id}.x", lower=-center_tol, upper=center_tol))
            constraints.append(interval(p.id, f"{A.id}.y", lower=-center_tol, upper=center_tol))

    return constraints
```

必须覆盖用户强调的规则：
> “XX 放在 YY 上方，那么 XX 的 z 必须大于 YY 的 z。”

实现上建议更严格：
```python
bottom(XX) >= top(YY) + margin_z
即 XX.z >= YY.z + YY.height + margin_z
```

如果只做 `XX.z > YY.z`，可能 XX 的底面仍穿进 YY 的体积。论文/系统里应采用 `bottom(XX) >= top(YY)` 作为强机械约束。

### 新增 `project_scene_to_constraints(scene, constraints)`

不要让 LLM 修坐标。用确定性投影：
```python
def project_scalar(value, lower=None, upper=None, target=None):
    if target is not None:
        return target
    if lower is not None and value < lower:
        return lower
    if upper is not None and value > upper:
        return upper
    return value
```

对每个 object 的 `x/y/z` 聚合所有约束：
```python
lower = max(all_lowers) if any else None
upper = min(all_uppers) if any else None
if lower is not None and upper is not None and lower > upper:
    return UNSAT(var, lower, upper, predicate_ids)
new_value = clamp_or_project(old_value, lower, upper)
```

对 `support(A,B)`：
1. 如果 `A.width > B.width - 2*inset` 或 `A.depth > B.depth - 2*inset`：
   - 优先缩小 A 的 width/depth 到 B 可支撑范围，记录 `support_resize`。
   - 如果物体不允许缩小，返回 `unsat_support_size`。
2. 设置：
   - `A.z = B.z + B.height`
   - `A.x` 限制到 B 的 depth 范围内
   - `A.y` 限制到 B 的 width 范围内

## 五、论文写作角度的核心卖点

当前项目如果按上述方案改造，论文方法部分可以从“LLM 生成 3D layout”升级为：

> A harnessed text-to-layout agent that converts natural-language spatial descriptions into machine-checkable scene predicates, compiles predicates into hard geometric constraints, projects generated layouts into the feasible set, and closes the loop with structured verification and repair logs before controllable text-to-image generation.

关键贡献表达：
- 不是简单 prompt engineering，而是 agent harness。
- 不是让模型“理解空间关系后输出坐标”，而是把空间语言降解为可计算谓词和硬约束。
- 不是人工看图调参，而是声明式规则 + 自动验证 + 失败驱动修复。
- 每一次失败都产生可读日志，可积累到跨会话知识库。

这会显著增强方法的科研品味：把问题从“LLM 坐标预测不准”提升为“自然语言空间控制的可验证中间表示和约束闭环”。

### 6.1 当前四步流程的目标职责

修改后的职责应是：

1. `agent_text2pkl_v5.py`：只负责从文字生成初始 scene graph、对象清单、谓词清单、初始 pkl，并立即调用机械约束工具把 pkl 投影到可行域。它不能只靠 prompt 让模型“自觉摆对”。
2. `agent_opinion.py`：从“自然语言评估 agent”升级为“机器验证器 + 报告器”。它输出 `validation.json`，其中每个失败项必须有 `predicate_id`、涉及对象、违反的 pkl 参数、期望取值范围、实际值、推荐修复工具。
3. `agent_reverse.py`：从“读 guidance 文本修 pkl”升级为“执行修复计划的工具调用器”。它读取 `validation.json` 或 `repair_plan.json`，按确定公式修改 pkl 参数，而不是让 LLM 再猜一次。
4. `infer2.py`：只在 pkl 的对象一致性、空间谓词、尺寸/体积、物理合理性、相机可见性全部通过后运行。不能用最终图像生成掩盖上游 pkl 错误。

推荐新增统一入口：

```bash
python3 run_scene_harness.py \
  --scene-text "..." \
  --output inference/saved_scenes/test.pkl \
  --placeholder-prompt "a photo of PLACEHOLDER ..." \
  --max-rounds 6
```

内部仍调用当前四个脚本，但流程由 `validation.json` 驱动，而不是人工决定跑几轮。

### 6.2 pkl/json 字段的硬约束总表

以 `inference/saved_scenes/example0.json` 为准，每个 subject 至少包含：

```json
{
  "name": "dog",
  "type": "dog",
  "dims": [width, depth, height],
  "x": [x_value],
  "y": [y_value],
  "z": [z_value],
  "azimuth": [radian],
  "bbox": [[0, 0, 0, 0]]
}
```

全局相机字段：

```json
{
  "camera_data": {
    "camera_elevation": 0.2059488517353309,
    "lens": 50.0,
    "global_scale": 1.0
  }
}
```

硬约束解释如下：

| 字段 | 含义 | 必须机械限制 |
|---|---|---|
| `subjects_data[*].name` | 对象实例名 | 必须与 prompt 主体一一对应，同类多个对象用稳定 id，如 `chair_1`、`chair_2` |
| `subjects_data[*].type` | 资产类型 | 必须属于 `asset_dimensions.json`，不能是未知类型；别名必须映射到 canonical type |
| `dims[0]` | width，左右宽度 | 必须在该 type 参考 width 的允许区间内，且满足体积/支撑面约束 |
| `dims[1]` | depth，前后深度 | 必须在该 type 参考 depth 的允许区间内，且满足体积/支撑面约束 |
| `dims[2]` | height，上下高度 | 必须在该 type 参考 height 的允许区间内，且满足体积/支撑面约束 |
| `x[0]` | 前后位置，增大为向前 | 由 front/behind/forward/backward 谓词或相机可见性工具约束 |
| `y[0]` | 左右位置，增大为向右 | 由 left/right 谓词约束 |
| `z[0]` | 底面高度，增大为向上 | 由 above/below/support/ground/flying 谓词约束 |
| `azimuth[0]` | 弧度制旋转 | 由 rotate/facing/aligned 谓词约束，角度输入必须转弧度 |
| `camera_data.global_scale` | 全局缩放 | 只用于整体视觉尺度和取景，不得用来修复对象间尺寸失真 |
| `camera_data.camera_elevation` | 相机俯仰，弧度 | 必须在可见性和视角范围内 |
| `camera_data.lens` | 焦距 | 必须在可见性范围内，优先调相机解决出画问题 |

所有硬约束必须进入 `constraints.json`，验证时逐条检查。任何违反都不能标记通过。

### 6.3 从 guidance.md 全量落实到机械参数限制

下面把 `guidance.md` 和 `代码说明.md` 中的规则改成必须实现的 pkl 参数公式。

#### 6.3.1 尺寸修改和尺寸区间

对每个对象 `A`：

```text
A.width  = A.dims[0]
A.depth  = A.dims[1]
A.height = A.dims[2]
A.volume = A.width * A.depth * A.height
```

如果 prompt/guidance 要求整体放大/缩小：

```text
new_dims = old_dims * multiplier
```

如果只要求 width/depth/height：

```text
new_width  = old_width  * multiplier
new_depth  = old_depth  * multiplier
new_height = old_height * multiplier
```

机械限制：

```text
ref_dims = asset_dimensions[type]
min_dim_i = ref_dims[i] * min_scale[type]
max_dim_i = ref_dims[i] * max_scale[type]
min_dim_i <= dims[i] <= max_dim_i
```

建议默认：

```text
普通物体：min_scale=0.70, max_scale=1.35
有 small/tiny/little 修饰：min_scale=0.45, max_scale=0.90
有 large/big/huge 修饰：min_scale=1.05, max_scale=1.80
支撑面上的小物体：width/depth 还必须小于支撑物可用台面
```

这一步解决 v5 中 cube 严重失真：不能出现同类型对象尺寸差异过大，也不能出现小物体体积超过大物体。

#### 6.3.2 绝对/相对平移

直接移动：

```text
move_forward(A, delta)  -> A.x = old_A.x + delta
move_backward(A, delta) -> A.x = old_A.x - delta
move_right(A, delta)    -> A.y = old_A.y + delta
move_left(A, delta)     -> A.y = old_A.y - delta
move_up(A, delta)       -> A.z = old_A.z + delta
move_down(A, delta)     -> A.z = old_A.z - delta
```

按参考物体尺寸移动：

```text
Move A forward by k times B.depth  -> A.x = old_A.x + k * B.dims[1]
Move A backward by k times B.depth -> A.x = old_A.x - k * B.dims[1]
Move A right by k times B.width    -> A.y = old_A.y + k * B.dims[0]
Move A left by k times B.width     -> A.y = old_A.y - k * B.dims[0]
Move A up by k times B.height      -> A.z = old_A.z + k * B.dims[2]
Move A down by k times B.height    -> A.z = old_A.z - k * B.dims[2]
```

这些是“赋值/更新公式”，不是 prompt 建议。修复日志必须记录 `old`、`new`、`delta`、`reference_object`、`multiplier`。

#### 6.3.3 相对位置关系的边界公式

定义边界：

```text
left(A)   = A.y - A.width / 2
right(A)  = A.y + A.width / 2
back(A)   = A.x - A.depth / 2
front(A)  = A.x + A.depth / 2
bottom(A) = A.z
top(A)    = A.z + A.height
```

非紧贴关系：

```text
left_of(A,B):      right(A) <= left(B) - margin_y
right_of(A,B):     left(A)  >= right(B) + margin_y
in_front_of(A,B):  back(A)  >= front(B) + margin_x
behind(A,B):       front(A) <= back(B) - margin_x
above(A,B):        bottom(A) >= top(B) + margin_z
below(A,B):        top(A) <= bottom(B) - margin_z
```

紧贴关系：

```text
tight_left_of(A,B):     right(A) = left(B)
tight_right_of(A,B):    left(A)  = right(B)
tight_in_front_of(A,B): back(A)  = front(B)
tight_behind(A,B):      front(A) = back(B)
tight_above(A,B):       bottom(A) = top(B)
tight_below(A,B):       top(A) = bottom(B)
```

用户强调的例子必须落实为：

```text
XX 放在 YY 上方:
XX.z >= YY.z + YY.dims[2] + margin_z
```

注意这比 `XX.z > YY.z` 更严格，因为当前 `z` 表示 cube 底面高度。只要求 `XX.z > YY.z` 仍可能穿模。

#### 6.3.4 支撑关系 support(A,B)

对 “A on B / A placed on B / A sits on B / A 放在 B 上 / countertop of B” 统一编译为：

```text
support(A,B)
```

硬约束：

```text
bottom(A) = top(B)
left(A)   >= left(B) + inset
right(A)  <= right(B) - inset
back(A)   >= back(B) + inset
front(A)  <= front(B) - inset
```

如果 A 太大放不下 B：

```text
if A.width > B.width - 2*inset:
    A.width = B.width - 2*inset
if A.depth > B.depth - 2*inset:
    A.depth = B.depth - 2*inset
```

如果缩小后违反该物体最小尺寸，返回：

```text
unsat_support_size(A,B)
```

这时不能让模型硬猜，应进入自动分歧处理：换支撑物、重新解析关系，或在低置信度时要求用户确认主体/关系。

支撑面槽位也要机械化：

```text
center_of_surface(A,B):      A.x = B.x, A.y = B.y
front_center_of_surface(A,B): A.x = B.x + max_offset_x, A.y = B.y
back_center_of_surface(A,B):  A.x = B.x - max_offset_x, A.y = B.y
front_left_of_surface(A,B):   A.x = B.x + max_offset_x, A.y = B.y - max_offset_y
front_right_of_surface(A,B):  A.x = B.x + max_offset_x, A.y = B.y + max_offset_y
back_left_of_surface(A,B):    A.x = B.x - max_offset_x, A.y = B.y - max_offset_y
back_right_of_surface(A,B):   A.x = B.x - max_offset_x, A.y = B.y + max_offset_y
```

其中：

```text
max_offset_x = max((B.depth - A.depth) / 2 - inset, 0)
max_offset_y = max((B.width - A.width) / 2 - inset, 0)
```

#### 6.3.5 组合位置关系

`front-left(A,B)`、`front-right(A,B)`、`back-left(A,B)`、`back-right(A,B)` 必须拆成两个谓词：

```text
front_left(A,B)  = in_front_of(A,B) + left_of(A,B)
front_right(A,B) = in_front_of(A,B) + right_of(A,B)
back_left(A,B)   = behind(A,B) + left_of(A,B)
back_right(A,B)  = behind(A,B) + right_of(A,B)
```

不能只调一个轴。

#### 6.3.6 地面、空中和物理合理性

默认地面约束：

```text
if not support(A,*) and not flying(A) and not hanging(A):
    A.z = 0
```

空中/飞行约束：

```text
flying(A): A.z >= min_flying_height
min_flying_height = max(0.5, reference_height(A))
```

非支撑物之间默认不能发生 3D 穿插。需要新增 `no_interpenetration(A,B)`：

```text
separated_x = front(A) <= back(B) - margin or front(B) <= back(A) - margin
separated_y = right(A) <= left(B) - margin or right(B) <= left(A) - margin
separated_z = top(A) <= bottom(B) - margin or top(B) <= bottom(A) - margin
no_interpenetration = separated_x or separated_y or separated_z
```

support(A,B) 是允许 z 接触但不允许水平 footprint 超出。

#### 6.3.7 旋转 azimuth

`azimuth` 是弧度制。prompt/guidance 通常是角度制，必须机械转换：

```text
radian_delta = degree_delta * pi / 180
counterclockwise: A.azimuth = old_A.azimuth + radian_delta
clockwise:        A.azimuth = old_A.azimuth - radian_delta
```

写入前规范化：

```text
A.azimuth = ((A.azimuth + pi) mod (2*pi)) - pi
```

如果 prompt 有 “same direction / parallel / aligned with B”：

```text
A.azimuth = B.azimuth
```

如果 prompt 有 “facing B”，可以先实现简化版：

```text
A.azimuth = atan2(B.y - A.y, B.x - A.x)
```

#### 6.3.8 相机参数

相机参数必须作为可见性工具的输出，而不是拿来修复语义错误。

硬范围建议：

```text
global_scale ∈ [0.4, 2.5]
camera_elevation_deg ∈ [5, 45]
lens_mm ∈ [18, 80]
```

如果 `verify_visibility` 发现 cube 出画：

1. 优先调整 `lens` 变小，扩大视野。
2. 再调整 `camera_elevation`。
3. 再整体平移/缩放布局。
4. 最后才允许轻微缩小物体，但缩小后必须重新跑尺寸/体积约束。

不能通过相机或 global_scale 掩盖 `left_of`、`support`、`above` 这类语义关系失败。

### 6.4 对象必须一一对应：不能多，不能少

`agent_text2pkl_v5.py` 必须新增 `object_manifest.json`：

```json
{
  "objects": [
    {
      "id": "refrigerator_1",
      "mention": "a refrigerator",
      "canonical_type": "refrigerator",
      "count": 1,
      "source_span": "there is a refrigerator",
      "confidence": 0.99
    }
  ]
}
```

机械检查：

```text
expected_ids = object_manifest.objects[*].id
actual_ids = subjects_data[*].id/name
expected_count_by_type == actual_count_by_type
expected_ids == actual_ids
```

规则：

- prompt 中 `a/an/one` 表示 1 个。
- 明确数字表示对应数量。
- `the` 默认指代已有对象，不新增对象。
- “countertop of cabinet” 中 `countertop` 是 cabinet 的部位，不新增 countertop，除非资产库中明确需要独立 countertop。
- “books on bookshelf” 如果 `books` 不是当前资产类型，必须进入 `unmapped_object`，不能静默丢弃。
- pkl 中不能出现 prompt 未提到的对象。
- prompt 中每个主体都必须在 pkl 中出现。

如果对象解析低置信度或无法映射资产类型，允许在生成前向用户确认一次：

```text
我解析到的主体是：refrigerator_1, cabinet_1, microwave_1, bowl_1。
其中 countertop 被解释为 cabinet 的支撑面，不作为独立物体。是否确认？
```

但确认之后，后续迭代循环必须全自动，不再依赖人工。

为了支持无人值守实验，可新增参数：

```text
--object-manifest path/to/object_manifest.json
--strict-object-match
--on-ambiguous-object fail|ask|best_effort
```

论文实验建议使用 `fail` 或预先提供 manifest，避免人工参与影响可复现性。

### 6.5 体积/尺寸相对关系工具

当前 v5 的 cube 大小失真，需要新增 `volume_constraint_tool.py`。

输入：

```text
subjects_data
asset_dimensions.json
object_scales.py
prompt size modifiers
```

计算：

```text
actual_volume(A) = A.width * A.depth * A.height
reference_volume(A) = ref_width(type) * ref_depth(type) * ref_height(type)
scale_ratio(A) = actual_volume(A) / reference_volume(A)
```

硬约束 1：单物体参考体积范围

```text
min_volume_ratio(type, modifier) <= scale_ratio(A) <= max_volume_ratio(type, modifier)
```

建议默认：

```text
普通物体：scale_ratio ∈ [0.45, 2.50]
small/tiny：scale_ratio ∈ [0.10, 0.75]
large/huge：scale_ratio ∈ [1.20, 5.80]
```

硬约束 2：跨物体体积顺序

对所有对象对 `(A,B)`：

```text
if reference_volume(A) > reference_volume(B) * order_margin:
    actual_volume(A) >= actual_volume(B) * min_order_ratio
```

建议：

```text
order_margin = 1.5
min_order_ratio = 1.15
```

例子：如果参考体积上 `car >> dog`，则实际 pkl 中必须满足：

```text
volume(car) > volume(dog)
```

如果违反，工具修复：

1. 优先把每个对象 dims 拉回参考 dims 区间。
2. 再按同一比例缩放失真的对象，使体积顺序满足。
3. 如果该对象有 `support(A,B)`，缩放后重新检查支撑 footprint。
4. 缩放后重新检查所有空间关系，因为 dims 改变会改变边界。

硬约束 3：同类型实例一致性

如果同一 prompt 中有多个同 type 且没有 size modifier：

```text
max(volume(A_i)) / min(volume(A_i)) <= 1.25
```

例如 two chairs 不能一个巨大、一个很小，除非 prompt 明确说 one large chair and one small chair。

输出：

```json
{
  "pass": false,
  "issues": [
    {
      "category": "volume_order_conflict",
      "object": "dog_1",
      "reference": "dog should be smaller than sedan",
      "actual": "volume(dog_1)=3.2, volume(sedan_1)=2.1",
      "repair_tool": "normalize_object_volume"
    }
  ]
}
```

### 6.6 自动验证 + 修复闭环，不依赖人工

新的自动循环：

```text
Round 0:
  agent_text2pkl_v5.py
    -> scene.pkl
    -> object_manifest.json
    -> predicates.json
    -> constraints.json

Loop:
  agent_opinion.py --json-output validation.json
  if validation.overall_pass:
      run infer2.py
      stop
  else:
      build repair_plan.json from validation.json
      agent_reverse.py --repair-plan repair_plan.json
      save repaired scene.pkl
      continue
```

`validation.json` 必须覆盖：

```json
{
  "overall_pass": false,
  "criteria": {
    "object_exact_match": {},
    "spatial_predicates": {},
    "support_predicates": {},
    "dimension_ranges": {},
    "volume_order": {},
    "collision_physics": {},
    "camera_visibility": {},
    "azimuth_constraints": {}
  },
  "issues": []
}
```

`repair_plan.json` 不能是自然语言 guidance，必须是可执行动作：

```json
{
  "actions": [
    {
      "tool": "set_param",
      "object": "microwave_1",
      "param": "z[0]",
      "value": 1.05,
      "reason": "support(microwave_1,cabinet_1): bottom(A)=top(B)"
    },
    {
      "tool": "clamp_param",
      "object": "cabinet_1",
      "param": "y[0]",
      "lower": 0.95,
      "upper": null,
      "reason": "right_of(cabinet_1,refrigerator_1)"
    }
  ]
}
```

`agent_reverse.py` 应新增：

```text
--repair-plan repair_plan.json
--constraints-json constraints.json
--object-manifest object_manifest.json
--predicates-json predicates.json
```

执行顺序固定为：

1. 对象增删/类型修复。
2. 尺寸和体积归一化。
3. 空间关系投影。
4. 支撑关系投影。
5. 碰撞消解。
6. 相机可见性调整。
7. 全量复验。

停止条件：

```text
pass -> run infer2.py
max_rounds reached -> 输出 failure_report.json，不运行 infer2.py
unsat constraints -> 输出 unsat_report.json，说明冲突谓词，不让 LLM 硬编
ambiguous objects with --on-ambiguous-object ask -> 向用户确认主体
```

这就是 harness 的关键：失败后不是“再让模型努力生成一次”，而是由验证信号决定调用哪个机械工具。

### 6.7 需要新增/改造的文件清单

最小实现可以不大改目录，先新增：

```text
harness_constraints.py
harness_object_manifest.py
harness_volume_tool.py
harness_validation.py
harness_repair_plan.py
run_scene_harness.py
```

并修改：

```text
agent_text2pkl_v5.py
agent_opinion.py
agent_reverse.py
```

具体改动：

- `agent_text2pkl_v5.py`
  - 输出 `object_manifest.json`、`predicates.json`、`constraints.json`。
  - 初始 pkl 生成后立即调用 constraint projector。
  - 默认启用 `--strict-object-match` 和 `--fail-on-unverified-relations`。

- `agent_opinion.py`
  - 不只输出中文修改建议。
  - 必须输出每个 pkl 参数违反的硬区间。
  - 必须输出可执行 `repair_plan.json`。

- `agent_reverse.py`
  - 新增 repair plan 模式。
  - 优先执行 JSON action，不再依赖自然语言 guidance。
  - 每次修改写 `repair_actions.jsonl`。

- `run_scene_harness.py`
  - 完全自动循环。
  - 只有所有验证通过才调用 `infer2.py`。

### 6.8 最小验收标准

对任意 prompt，至少必须通过以下检查：

1. prompt 主体和 pkl subject 一一对应，不能多、不能少。
2. 每个 subject 的 `type` 都在资产库中。
3. 每个 subject 的 `dims` 在参考尺寸和体积范围内。
4. 所有 prompt 空间关系都已转换为 predicates。
5. 每个 predicate 都有对应 pkl 参数约束。
6. `above/below/left/right/front/behind/support/tight_*` 全部按边界公式验证。
7. 所有非支撑物不穿模。
8. 所有 cube 完整可见。
9. 失败时自动生成 `repair_plan.json` 并由 `agent_reverse.py` 执行。
10. 达到最大轮数仍失败时，不运行 `infer2.py`，而是输出结构化失败报告。

这套补充后，方案才真正满足用户要求：空间位置关系、尺寸体积关系、对象一一对应和修复闭环都由 pkl 参数上的强制机械规则保证，而不是依赖 prompt 让模型记住。

## 七、逐条 pkl 参数机械化规格：覆盖 dims / x / y / z / azimuth / camera / 增删物体

本节把用户列出的每一类 guidance 全部落到 pkl/json 原始字段上。目标不是“让 prompt 更详细”，而是新增工具或脚本，把自然语言关系编译成 `subjects_data[*]` 和 `camera_data` 的硬约束、硬赋值或硬更新公式。

### 7.1 必须新增的工具接口

建议新增一个统一工具：

```text
harness_param_constraints.py
```

核心接口：

```python
def parse_param_directives(scene_text, object_manifest) -> list[Directive]:
    """从 prompt/guidance 中抽取尺寸、平移、旋转、相机、增删物体、相对位置操作。"""

def compile_directive_to_constraints(scene, directive) -> list[ConstraintAction]:
    """把每条 Directive 编译成 pkl 字段级约束或赋值动作。"""

def apply_constraint_actions(scene, actions) -> tuple[scene, repair_log]:
    """确定性修改 pkl 字段，不调用 LLM 猜坐标。"""

def verify_constraint_actions(scene, actions) -> ValidationReport:
    """逐条验证 pkl 字段是否满足约束。"""
```

每个 `ConstraintAction` 必须至少包含：

```json
{
  "source_text": "Move chair left by 1.2 times the width of table.",
  "target_object": "chair_1",
  "field": "subjects_data[chair_1].y[0]",
  "mode": "set | add | multiply | clamp | interval | delete | create",
  "value": null,
  "lower": null,
  "upper": null,
  "formula": "old_y - 1.2 * table_1.dims[0]",
  "reason": "move_left_by_reference_width(chair_1, table_1, 1.2)"
}
```

后续 `agent_reverse.py` 不应再消费自然语言 guidance，而应优先消费这种 JSON action。

### 7.2 pkl 字段读取规范

对 `inference/saved_scenes/example0.json` 这类文件，统一用以下方式读取：

```text
A.width   = A.dims[0]
A.depth   = A.dims[1]
A.height  = A.dims[2]
A.x       = A.x[0]
A.y       = A.y[0]
A.z       = A.z[0]
A.azimuth = A.azimuth[0]
```

写回时必须保持原格式：

```text
subjects_data[*].dims = [width, depth, height]
subjects_data[*].x = [x]
subjects_data[*].y = [y]
subjects_data[*].z = [z]
subjects_data[*].azimuth = [radian]
subjects_data[*].bbox = [[0, 0, 0, 0]]
```

禁止工具内部混用 `scene_x` 和 pkl `x` 而不记录转换。如果使用 `agent_text2pkl_v5.py` 内部的 `scene_x`，必须显式记录：

```text
pkl_x = scene_x - 6.0
scene_x = pkl_x + 6.0
```

约束计算建议直接在 pkl 原始坐标上做，减少偏移错误。

### 7.3 dims：Width / Depth / Height 的机械化规则

#### 7.3.1 维度含义

```text
dims[0] = width  = 左右宽度，对应 y 方向占用
dims[1] = depth  = 前后深度，对应 x 方向占用
dims[2] = height = 上下高度，对应 z 方向占用
```

#### 7.3.2 单维度增大/减小

输入语义：

```text
Increase / decrease XX along the width dimension
Increase / decrease XX along the depth dimension
Increase / decrease XX along the height dimension
增大/减小 XX 的 width/depth/height
```

机械动作：

```text
increase_width(A, k):  A.dims[0] = old_A.dims[0] * k
decrease_width(A, k):  A.dims[0] = old_A.dims[0] * k
increase_depth(A, k):  A.dims[1] = old_A.dims[1] * k
decrease_depth(A, k):  A.dims[1] = old_A.dims[1] * k
increase_height(A, k): A.dims[2] = old_A.dims[2] * k
decrease_height(A, k): A.dims[2] = old_A.dims[2] * k
```

其中：

```text
如果文本明确 “变为现在的 0.5 倍”，k=0.5
如果文本明确 “增加为现在的 1.2 倍”，k=1.2
如果只写 “增大/放大”，默认 k=1.12
如果只写 “减小/缩小”，默认 k=0.88
```

后置硬约束：

```text
min_dim(A.type, i) <= A.dims[i] <= max_dim(A.type, i)
A.dims[i] > 0
```

默认范围：

```text
min_dim(A.type, i) = asset_dimensions[A.type][i] * 0.70
max_dim(A.type, i) = asset_dimensions[A.type][i] * 1.35
```

如果 prompt 中有 `small/tiny/little/mini`，范围改为：

```text
min = ref * 0.45
max = ref * 0.90
```

如果 prompt 中有 `large/big/huge/giant`，范围改为：

```text
min = ref * 1.05
max = ref * 1.80
```

如果对象有 `support(A,B)`，还必须满足：

```text
A.dims[0] <= B.dims[0] - 2 * inset
A.dims[1] <= B.dims[1] - 2 * inset
```

#### 7.3.3 所有维度一起增大/减小

输入语义：

```text
Increase all dimensions of XX
Decrease all dimensions of XX
所有维度增大/减小
放大/缩小 XX
```

机械动作：

```text
A.dims[0] = old_A.dims[0] * k
A.dims[1] = old_A.dims[1] * k
A.dims[2] = old_A.dims[2] * k
```

验证：

```text
∀i in {0,1,2}: min_dim(A.type,i) <= A.dims[i] <= max_dim(A.type,i)
```

注意：修改 dims 后必须重新验证所有空间关系，因为边界 `left/right/front/back/top` 会随 dims 改变。

### 7.4 x：前后平移的机械化规则

字段：

```text
A.x = subjects_data[A].x[0]
```

方向：

```text
x 增大 -> 向前 / closer to viewer / forward
x 减小 -> 向后 / farther from viewer / backward
```

#### 7.4.1 直接向前/向后

如果文本只说 “向前/向后移动一点”，默认：

```text
default_delta_x(A) = max(0.25, A.dims[1] * 0.5)
```

机械动作：

```text
Move A forward:  A.x = old_A.x + default_delta_x(A)
Move A backward: A.x = old_A.x - default_delta_x(A)
```

#### 7.4.2 按参考物体 Depth 平移

输入语义：

```text
XX 向前/向后平移 0.7 个 YY 的 Depth
Move XX forward/backward by 0.7 times the depth of YY
```

机械动作：

```text
delta_x = multiplier * YY.dims[1]
forward:  XX.x = old_XX.x + delta_x
backward: XX.x = old_XX.x - delta_x
```

必须记录：

```json
{
  "field": "XX.x[0]",
  "formula": "old_XX.x + 0.7 * YY.dims[1]",
  "reference_field": "YY.dims[1]"
}
```

后置验证：

```text
abs(new_XX.x - expected_x) <= eps
```

### 7.5 y：左右平移的机械化规则

字段：

```text
A.y = subjects_data[A].y[0]
```

方向：

```text
y 减小 -> 向左 / left
y 增大 -> 向右 / right
```

#### 7.5.1 直接向左/向右

默认步长：

```text
default_delta_y(A) = max(0.25, A.dims[0] * 0.5)
```

机械动作：

```text
Move A left:  A.y = old_A.y - default_delta_y(A)
Move A right: A.y = old_A.y + default_delta_y(A)
```

#### 7.5.2 按参考物体 Width 平移

输入语义：

```text
XX 向左/向右平移 1.2 个 YY 的 Width
Move XX left/right by 1.2 times the width of YY
```

机械动作：

```text
delta_y = multiplier * YY.dims[0]
left:  XX.y = old_XX.y - delta_y
right: XX.y = old_XX.y + delta_y
```

验证：

```text
abs(new_XX.y - expected_y) <= eps
```

### 7.6 z：上下平移的机械化规则

字段：

```text
A.z = subjects_data[A].z[0]
```

方向：

```text
z 增大 -> 向上 / up
z 减小 -> 向下 / down
```

#### 7.6.1 直接向上/向下

默认步长：

```text
default_delta_z(A) = max(0.15, A.dims[2] * 0.5)
```

机械动作：

```text
Move A up:   A.z = old_A.z + default_delta_z(A)
Move A down: A.z = max(0, old_A.z - default_delta_z(A))
```

#### 7.6.2 按参考物体 Height 平移

输入语义：

```text
XX 向上/向下平移 0.5 个 YY 的 Height
Move XX up/down by 0.5 times the height of YY
```

机械动作：

```text
delta_z = multiplier * YY.dims[2]
up:   XX.z = old_XX.z + delta_z
down: XX.z = max(0, old_XX.z - delta_z)
```

如果该对象有 `support(XX, S)`，手动上下移动会破坏支撑关系。此时必须二选一：

```text
1. 如果移动语义优先：移除 support(XX,S)，并重新验证物理合理性
2. 如果支撑语义优先：拒绝上下移动，报告 conflict(move_z, support)
```

### 7.7 azimuth：旋转角度机械化规则

字段：

```text
A.azimuth = subjects_data[A].azimuth[0]
```

角度转换：

```text
degree = radian * 180 / pi
radian = degree * pi / 180
```

输入语义：

```text
Rotate A by D degrees
A 顺时针/逆时针旋转 D 度
```

机械动作：

```text
counterclockwise: A.azimuth = old_A.azimuth + D * pi / 180
clockwise:        A.azimuth = old_A.azimuth - D * pi / 180
unspecified:      A.azimuth = old_A.azimuth + D * pi / 180
```

写回前归一化：

```text
A.azimuth = ((A.azimuth + pi) % (2*pi)) - pi
```

验证：

```text
abs(normalize_angle(new_A.azimuth - expected_azimuth)) <= eps_angle
```

### 7.8 camera_data：相机参数机械化规则

#### 7.8.1 global_scale

字段：

```text
camera_data.global_scale
```

输入语义：

```text
Scale all cube up/down
所有物体整体放大/放小
```

机械动作：

```text
scale_up:   global_scale = old_global_scale * k
scale_down: global_scale = old_global_scale * k
```

默认：

```text
scale_up 默认 k=1.10
scale_down 默认 k=0.90
```

硬范围：

```text
0.4 <= global_scale <= 2.5
```

注意：`global_scale` 只影响整体显示尺度，不允许用它修复“dog 比 car 大”这种对象间尺寸错误。对象间尺寸错误必须改 `dims`。

#### 7.8.2 camera_elevation

字段：

```text
camera_data.camera_elevation  # radian
```

输入语义：

```text
相机俯仰增大/减小
camera elevation increase/decrease
```

机械动作：

```text
new_elevation_rad = old_elevation_rad + delta_degree * pi / 180
```

硬范围：

```text
5° <= camera_elevation <= 45°
即 5*pi/180 <= camera_data.camera_elevation <= 45*pi/180
```

#### 7.8.3 lens

字段：

```text
camera_data.lens
```

输入语义：

```text
相机焦距增大/减小 10
increase/decrease lens by 10
```

机械动作：

```text
increase_lens(delta): lens = old_lens + delta
decrease_lens(delta): lens = old_lens - delta
```

硬范围：

```text
18 <= lens <= 80
```

可见性修复优先级：

```text
frame_clipping -> 先 decrease lens -> 再调 elevation -> 再调 global_scale -> 最后才允许缩小 dims
```

### 7.9 相对空间关系：全部转换为边界区间

定义边界：

```text
left(A)   = A.y - A.dims[0] / 2
right(A)  = A.y + A.dims[0] / 2
back(A)   = A.x - A.dims[1] / 2
front(A)  = A.x + A.dims[1] / 2
bottom(A) = A.z
top(A)    = A.z + A.dims[2]
```

建议：

```text
margin_x = max(0.05, 0.05 * max(A.dims[1], B.dims[1]))
margin_y = max(0.05, 0.05 * max(A.dims[0], B.dims[0]))
margin_z = 0.02
inset = 0.02
```

#### 7.9.1 XX 放在 YY 上方

输入语义：

```text
XX 放在 YY 上方/上面/顶部
Place XX above YY / on top of YY
```

如果是普通 `above`：

```text
XX.x ∈ [YY.x - YY.dims[1]/2, YY.x + YY.dims[1]/2]
XX.y ∈ [YY.y - YY.dims[0]/2, YY.y + YY.dims[0]/2]
XX.z >= YY.z + YY.dims[2] + margin_z
```

如果是 `on / on top of / sits on / placed on / 放在...上`，应升级为 `support(XX,YY)`：

```text
XX.z = YY.z + YY.dims[2]
left(XX) >= left(YY) + inset
right(XX) <= right(YY) - inset
back(XX) >= back(YY) + inset
front(XX) <= front(YY) - inset
```

#### 7.9.2 XX 放在 YY 下方

```text
XX.x ∈ [YY.x - YY.dims[1]/2, YY.x + YY.dims[1]/2]
XX.y ∈ [YY.y - YY.dims[0]/2, YY.y + YY.dims[0]/2]
XX.z + XX.dims[2] <= YY.z - margin_z
```

若 `YY.z = 0` 导致 `XX.z < 0`，则该约束不可满足，输出：

```text
unsat_below_ground(XX,YY)
```

不能强行把 XX 放到地面以下。

#### 7.9.3 XX 放在 YY 左边 / 右边

左边：

```text
right(XX) <= left(YY) - margin_y
等价：XX.y <= YY.y - YY.dims[0]/2 - XX.dims[0]/2 - margin_y
```

右边：

```text
left(XX) >= right(YY) + margin_y
等价：XX.y >= YY.y + YY.dims[0]/2 + XX.dims[0]/2 + margin_y
```

如果没有同时指定前后关系，为保持对齐：

```text
XX.x = YY.x
```

#### 7.9.4 XX 放在 YY 前面 / 后面

前面：

```text
back(XX) >= front(YY) + margin_x
等价：XX.x >= YY.x + YY.dims[1]/2 + XX.dims[1]/2 + margin_x
```

后面：

```text
front(XX) <= back(YY) - margin_x
等价：XX.x <= YY.x - YY.dims[1]/2 - XX.dims[1]/2 - margin_x
```

如果没有同时指定左右关系，为保持对齐：

```text
XX.y = YY.y
```

#### 7.9.5 XX 紧贴在 YY 上方 / 下方

紧贴上方：

```text
bottom(XX) = top(YY)
即 XX.z = YY.z + YY.dims[2]
```

紧贴下方：

```text
top(XX) = bottom(YY)
即 XX.z = YY.z - XX.dims[2]
```

同时要求 x/y 水平投影在 YY 面积内：

```text
XX.x ∈ [YY.x - YY.dims[1]/2, YY.x + YY.dims[1]/2]
XX.y ∈ [YY.y - YY.dims[0]/2, YY.y + YY.dims[0]/2]
```

#### 7.9.6 XX 紧贴在 YY 左边 / 右边

紧贴左边：

```text
right(XX) = left(YY)
XX.y = YY.y - YY.dims[0]/2 - XX.dims[0]/2
```

紧贴右边：

```text
left(XX) = right(YY)
XX.y = YY.y + YY.dims[0]/2 + XX.dims[0]/2
```

默认对齐：

```text
XX.x = YY.x
```

#### 7.9.7 XX 紧贴在 YY 前面 / 后面

紧贴前面：

```text
back(XX) = front(YY)
XX.x = YY.x + YY.dims[1]/2 + XX.dims[1]/2
```

紧贴后面：

```text
front(XX) = back(YY)
XX.x = YY.x - YY.dims[1]/2 - XX.dims[1]/2
```

默认对齐：

```text
XX.y = YY.y
```

### 7.10 增加 / 删除物体的机械化规则

#### 7.10.1 增加物体

输入语义：

```text
增加物体 XX
Add XX
```

机械动作：

```json
{
  "name": "XX_1",
  "type": "canonical_type",
  "dims": "asset_dimensions[canonical_type] * object_scales[canonical_type]",
  "x": [initial_x],
  "y": [initial_y],
  "z": [0],
  "azimuth": [0],
  "bbox": [[0, 0, 0, 0]]
}
```

初始位置不能随机猜，必须由关系决定：

```text
如果有 relation(XX,YY): 用该 relation 的公式求 x/y/z
如果没有 relation: 放入 next_free_slot，并记录 missing_relation_warning
```

`next_free_slot` 必须通过碰撞检测选择：

```text
候选点按半径扩展生成
选择第一个满足 no_interpenetration 且 camera_visible 的位置
```

#### 7.10.2 删除物体

输入语义：

```text
删除物体 XX
Delete XX
```

机械动作：

```text
subjects_data = [s for s in subjects_data if s.id/name != XX]
object_manifest 中也删除 XX
predicates 中删除所有涉及 XX 的谓词
constraints 中删除所有涉及 XX 的约束
```

删除后必须重新验证：

```text
object_exact_match
all remaining predicates
collision
camera_visibility
```

### 7.11 prompt 主体一一对应：输入阶段就强制明确

为了保证“prompt 中每个物体 pkl 必须有，不能多不能少”，建议把 `agent_text2pkl_v5.py` 的输入从只有 `--scene-text` 扩展为：

```bash
python3 agent_text2pkl_v5.py \
  --scene-text "..." \
  --objects "refrigerator:1,cabinet:1,microwave:1,bowl:1" \
  --strict-object-match \
  --output ...
```

或使用 JSON：

```bash
python3 agent_text2pkl_v5.py \
  --scene-text "..." \
  --object-manifest input_objects.json \
  --strict-object-match \
  --output ...
```

`input_objects.json`：

```json
{
  "objects": [
    {"id": "refrigerator_1", "type": "refrigerator", "mention": "a refrigerator"},
    {"id": "cabinet_1", "type": "cabinet", "mention": "a cabinet"},
    {"id": "microwave_1", "type": "microwave", "mention": "a microwave"},
    {"id": "bowl_1", "type": "bowl", "mention": "a bowl"}
  ],
  "non_objects": [
    {"mention": "countertop", "reason": "part_of(cabinet_1), not standalone asset"}
  ]
}
```

如果用户没有提供主体清单，`agent_text2pkl_v5.py` 必须先生成候选清单并进入确认/失败机制：

```text
--on-ambiguous-object ask:
  向用户确认主体清单

--on-ambiguous-object fail:
  输出 object_parse_failure.json，不生成 pkl

--on-ambiguous-object best_effort:
  自动选择最高置信度主体，但必须在 validation.json 中标记 non_reproducible_warning
```

严格验证：

```text
expected = sorted(object_manifest.objects[*].id)
actual = sorted(subjects_data[*].name 或 subjects_data[*].id)
expected == actual
```

类型数量验证：

```text
Counter(object_manifest.objects[*].type) == Counter(subjects_data[*].type)
```

禁止行为：

```text
1. prompt 未提到的物体不能出现在 pkl
2. prompt 提到的主体不能被静默丢弃
3. the 不能当作新增物体
4. 部位词 countertop / surface / top / side 默认不能当独立物体，除非 manifest 明确列为 object
```

### 7.12 agent_text2pkl_v5.py 的强制执行顺序

`agent_text2pkl_v5.py` 修改后必须按以下顺序执行：

```text
1. parse_or_load_object_manifest()
2. strict_validate_manifest_against_scene_text()
3. parse_predicates(scene_text, object_manifest)
4. create_initial_subjects_from_manifest()
5. normalize_dims_by_asset_reference()
6. compile_param_constraints_for_all_directives()
7. apply_param_constraints()
8. verify_object_exact_match()
9. verify_all_param_constraints()
10. verify_volume_and_dims()
11. verify_collision_and_visibility()
12. save pkl/json + manifest/predicates/constraints/validation/repair_log
```

只要第 8 到第 11 任一步失败，就不能把 pkl 当成最终结果。必须进入自动修复循环，或在不可满足时输出 `unsat_report.json`。

### 7.13 本节验收用例

用例 1：

```text
XX 放在 YY 上方
```

必须生成：

```text
XX.z >= YY.z + YY.dims[2] + margin_z
```

用例 2：

```text
XX 紧贴在 YY 右边
```

必须生成：

```text
XX.y = YY.y + YY.dims[0]/2 + XX.dims[0]/2
XX.x = YY.x
```

用例 3：

```text
XX 向左平移 1.2 个 YY 的 Width
```

必须生成：

```text
XX.y = old_XX.y - 1.2 * YY.dims[0]
```

用例 4：

```text
减小 desk 的 Height 为现在的 0.5 倍
```

必须生成：

```text
desk.dims[2] = old_desk.dims[2] * 0.5
```

用例 5：

```text
Rotate chair clockwise by 90 degrees
```

必须生成：

```text
chair.azimuth[0] = normalize(old_chair.azimuth[0] - 90*pi/180)
```

用例 6：

```text
prompt 主体: refrigerator, cabinet, microwave, bowl
```

必须验证：

```text
subjects_data 中只有 refrigerator_1, cabinet_1, microwave_1, bowl_1
不能多，不能少
```

这一章是对要求 A/B 的直接落地规格。实现后，`agent_text2pkl_v5.py` 的忠实度不再依赖模型是否“记住规则”，而依赖脚本是否能把文字关系编译成 pkl 参数上的可检查约束。
